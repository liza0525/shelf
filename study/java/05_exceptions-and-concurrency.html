Checked vs Unchecked

Python의 예외는 전부 unchecked에 가깝습니다. Java는 "호출자가 반드시 처리해야 하는 예외"를 컴파일러가 강제한다는 점이 다릅니다.

Checked Exception

  • Exception을 상속 (RuntimeException 제외)
  • 메서드 시그니처에 throws로 명시해야 함
  • 호출자가 catch 하거나 다시 throws 선언
  • 예: IOException, SQLException
  • Python에는 대응 개념이 없음

Unchecked Exception

  • RuntimeException을 상속
  • throws 선언 없이도 던질 수 있음
  • 컴파일러가 처리를 강제하지 않음
  • 예: NullPointerException, IllegalArgumentException
  • Python의 예외와 동작 방식이 가장 비슷함

try-with-resources

AutoCloseable을 구현한 자원은 try() 괄호 안에서 선언하면 블록을 벗어날 때 자동으로 close()가 호출됩니다.

여러 리소스를 안전하게 정리하기

public class ResourceExample { public static void main(String[] args) { // 세미콜론으로 여러 리소스를 나열 가능 try (BufferedReader br = new BufferedReader(new FileReader("data.txt")); FileWriter fw = new FileWriter("out.txt")) { String line = br.readLine(); fw.write(line); } catch (IOException e) { // close()는 선언의 역순으로 자동 호출됨 (fw -> br) System.err.println("파일 처리 실패: " + e.getMessage()); } } }

Python의 with open(...) as f: 와 목적은 같지만, 리소스를 여러 개 나열하고 예외를 명시적으로 catch해야 한다는 점이 다릅니다.

커스텀 예외 설계

호출자가 복구 가능한 상황만 checked로, 나머지는 프로그래밍 오류로 보고 unchecked(RuntimeException)로 설계하는 것이 관례입니다.

도메인 의미를 담은 checked 예외

public class InsufficientBalanceException extends Exception { private final BigDecimal shortage; public InsufficientBalanceException(BigDecimal shortage) { super("잔액이 " + shortage + "원 부족합니다."); this.shortage = shortage; } public BigDecimal getShortage() { return shortage; } } // throws로 계약을 명시하면 호출부가 반드시 처리하게 됨 public void withdraw(BigDecimal amount) throws InsufficientBalanceException { if (amount.compareTo(balance) > 0) { throw new InsufficientBalanceException(amount.subtract(balance)); } balance = balance.subtract(amount); }

Thread와 Runnable

Thread를 상속하는 방식보다 Runnable을 구현하는 방식이 권장됩니다. 상속 슬롯을 다른 용도로 남겨둘 수 있기 때문입니다.

start()는 새 스레드, run()은 그냥 메서드 호출

// 1) Thread 상속 방식 class DownloadThread extends Thread { @Override public void run() { System.out.println("다운로드 중... " + Thread.currentThread().getName()); } } // 2) Runnable 구현 방식 (더 권장됨) Runnable task = () -> { System.out.println("작업 실행: " + Thread.currentThread().getName()); }; Thread t1 = new DownloadThread(); Thread t2 = new Thread(task); t1.start(); // run()이 아니라 start()를 호출해야 새 스레드가 생성됨 t2.start(); t1.join(); // 스레드가 끝날 때까지 대기 t2.join();

synchronized와 volatile

synchronized는 상호 배제(한 번에 한 스레드만 진입)와 가시성을 모두 보장하고, volatile은 가시성만 보장합니다. 용도가 다릅니다.

데이터 레이스 예방

class Counter { private int count = 0; // count++는 read-modify-write 3단계라 원자적이지 않음 → 값 유실 가능 public void incrementUnsafe() { count++; } // synchronized: 한 번에 한 스레드만 이 메서드에 진입 (모니터 락) public synchronized void incrementSafe() { count++; } } class FlagHolder { // volatile: 모든 스레드가 항상 메인 메모리에서 최신 값을 읽도록 보장 // (가시성만 보장, 복합 연산의 원자성은 보장하지 않음) private volatile boolean running = true; public void stop() { running = false; } }

GIL이 없다면

Python 스레드는 CPU-bound 작업에서 사실상 순차 실행됩니다. Java 스레드는 코어마다 진짜 병렬로 돌아갑니다.

Python

  • GIL(Global Interpreter Lock) 때문에 한 번에 바이트코드 1개만 실행
  • 스레드를 여러 개 만들어도 CPU-bound 코드는 병렬 아님
  • 진짜 병렬 처리를 하려면 multiprocessing 필요
  • GIL 덕분에 인터프리터 내부 상태는 항상 안전

Java

  • 스레드 하나 = 진짜 OS 스레드, 멀티코어를 그대로 활용
  • 전역 락(GIL) 자체가 없음
  • 대신 개발자가 synchronized/volatile/Lock으로 직접 동기화해야 함
  • 데이터 레이스는 컴파일러가 안 막아주고 런타임에만 드러남

ExecutorService 스레드 풀

new Thread()를 매번 만드는 대신, 재사용 가능한 스레드 풀에 작업을 제출합니다. 생성/소멸 비용을 줄이고 동시 실행 개수도 제어할 수 있습니다.

invokeAll로 여러 작업 동시 실행

ExecutorService pool = Executors.newFixedThreadPool(4); List<Callable<String>> tasks = List.of( () -> fetchUrl("https://a.example.com"), () -> fetchUrl("https://b.example.com"), () -> fetchUrl("https://c.example.com") ); // invokeAll: 모든 작업을 제출하고 전부 끝날 때까지 블록 List<Future<String>> results = pool.invokeAll(tasks); for (Future<String> f : results) { System.out.println(f.get()); // 결과 또는 예외 확인 } pool.shutdown(); // 새 작업은 거부, 기존 작업은 계속 진행 후 종료

CompletableFuture 조합

Future는 결과를 블로킹으로만 꺼낼 수 있지만, CompletableFuture는 콜백을 체이닝해서 비동기 파이프라인을 구성할 수 있습니다.

thenCombine / thenAccept / exceptionally

CompletableFuture<String> userFuture = CompletableFuture.supplyAsync(() -> fetchUser(id)); CompletableFuture<String> orderFuture = CompletableFuture.supplyAsync(() -> fetchOrders(id)); // 두 비동기 작업이 모두 끝나면 결과를 합쳐 다음 단계로 CompletableFuture<String> combined = userFuture.thenCombine( orderFuture, (user, orders) -> user + " - " + orders ); combined.thenAccept(System.out::println) .exceptionally(ex -> { System.err.println("실패: " + ex.getMessage()); return null; }); combined.join(); // 필요할 때만 결과를 기다림

Python asyncio.gather와 개념은 비슷하지만, CompletableFuture는 스레드 풀 위에서 진짜 병렬로 실행됩니다.

흔한 실수

01

Checked Exception을 조용히 삼키기 catch (Exception e) {} 처럼 빈 블록으로 무시하면 실패가 흔적도 없이 사라집니다. 최소한 로깅이라도 남기세요.

02

데드락(deadlock) 두 스레드가 두 개의 락을 서로 다른 순서로 획득하려 하면 상대방이 락을 놓기를 영원히 기다리며 멈춥니다.

03

공유 변수에 synchronized/volatile 빠뜨리기 필드 하나만 동기화를 빼먹어도 데이터 레이스가 발생합니다. 특히 count++ 같은 read-modify-write 연산은 반드시 확인하세요.

04

ExecutorService.shutdown()을 호출하지 않기 풀을 닫지 않으면 스레드가 계속 살아있어 JVM 프로세스가 종료되지 않는 경우가 생깁니다.
