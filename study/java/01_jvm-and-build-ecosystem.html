JVM과 빌드 생태계

← → 키로 페이지를 넘길 수 있어요.

JVM 동작 방식

.java가 실행되기까지 5단계

Python은 python 명령 한 번으로 컴파일과 실행이 뒤섞여 일어나지만, Java는 컴파일(javac)과 실행(java)이 완전히 분리된 두 단계예요.

STEP 1

javac 컴파일

.java 소스를 바이트코드로 변환

STEP 2

.class 파일

JVM 전용 중간 표현, 플랫폼 독립적

STEP 3

Class Loader

실행 시점에 .class를 JVM 메모리로 적재

STEP 4

인터프리터 + JIT

처음엔 해석 실행, hot 코드는 네이티브로 재컴파일

STEP 5

실행 & GC

Heap 객체는 GC 스레드가 별도로 회수

핵심 개념

바이트코드 · 클래스 로더 · JIT · GC

BYTECODE

.class 파일

javac가 생성하는 JVM 전용 명령어 집합(opcode). 사람이 아니라 JVM이 읽는 저수준 표현이라 "한 번 컴파일하면 어디서나 실행"이 가능하다.

CLASS LOADER

동적 적재

클래스는 컴파일 시점이 아니라 실제로 필요해지는 실행 시점에 로드된다. Bootstrap → Platform → Application 순으로 위임(delegation)하며 찾는다.

JIT

Just-In-Time 컴파일

처음엔 인터프리터가 바이트코드를 한 줄씩 해석하다가, 반복 실행되는 hot 메서드를 감지해 네이티브 머신 코드로 즉석 컴파일하고 캐싱한다.

GC

가비지 컬렉션

Heap에서 더 이상 도달 불가능해진 객체를 GC 스레드가 자동으로 회수한다. 기본 전략은 세대별(Young/Old) 수집이며, 참조 카운팅은 쓰지 않는다.

비교

CPython과 뭐가 다를까

CPython

  • .py 실행마다 즉석에서 바이트코드(.pyc)로 컴파일
  • 레퍼런스 카운팅으로 즉시 회수 + 순환 참조용 보조 GC
  • 기본 인터프리터는 JIT 없이 바이트코드를 그대로 해석
  • GIL 때문에 한 프로세스 안에서 진짜 병렬 실행 불가

JVM

  • javac가 배포 전에 미리 바이트코드(.class)로 컴파일
  • 세대별 GC가 도달 불가능한 객체를 주기적으로 일괄 회수
  • JIT가 hot path를 감지해 네이티브 코드로 최적화
  • 스레드가 OS 네이티브 스레드라 GIL 없이 진짜 병렬 실행 가능

빌드 도구

Maven · Gradle · pip/poetry

셋 다 "의존성을 어떻게 선언하고, 어떻게 재현 가능하게 설치하는가"를 해결하려는 도구예요. 문법만 다를 뿐 역할은 거의 대응된다.

MAVEN

pom.xml (XML)

선언적 설정. 정해진 라이프사이클(compile → test → package)을 그대로 따르는 게 기본이라, 관례를 따르면 설정이 짧아진다.

GRADLE

build.gradle (DSL)

빌드 스크립트를 코드로 작성해 태스크를 유연하게 커스터마이징. 증분 빌드와 캐싱 덕분에 Maven보다 빠른 경우가 많다.

PIP / POETRY

requirements.txt / pyproject.toml

pip은 설치만, poetry는 의존성 해석 + lock + 가상환경까지. Maven/Gradle의 좌표 기반 의존성 관리가 이 lock 파일과 가장 가깝다.

Maven

pom.xml 구조

groupId/artifactId/version이 좌표(coordinate)가 되고, <dependencies>가 Python의 requirements.txt 역할을 해요.

pom.xml

<!-- 프로젝트 좌표: groupId + artifactId + version --> <project> <groupId>com.study</groupId> <artifactId>hello-world</artifactId> <version>1.0.0</version> <properties> <maven.compiler.release>21</maven.compiler.release> </properties> <!-- 의존성: Maven Central에서 좌표로 자동 다운로드 --> <dependencies> <dependency> <groupId>org.junit.jupiter</groupId> <artifactId>junit-jupiter</artifactId> <version>5.10.0</version> <scope>test</scope> </dependency> </dependencies> </project>

Gradle

build.gradle 구조

같은 의존성을 Gradle은 DSL(Groovy 기준)로 더 짧게 표현해요. Kotlin DSL(build.gradle.kts)을 쓰는 프로젝트도 많아요.

build.gradle

// 플러그인: 어떤 종류의 프로젝트인지 선언 plugins { id 'java' id 'application' } group = 'com.study' version = '1.0.0' java { toolchain { languageVersion = JavaLanguageVersion.of(21) } } // 의존성: Maven Central 좌표를 그대로 사용 repositories { mavenCentral() } dependencies { testImplementation 'org.junit.jupiter:junit-jupiter:5.10.0' }

타입 시스템

컴파일 타임에 다 걸러진다

Python은 실행해봐야 TypeError를 만나지만, Java는 javac 단계에서 타입 불일치를 전부 잡아내요. var는 타입을 "생략"하는 게 아니라 컴파일러가 "추론"하는 거예요.

TypeCheck.java

void greet(String name) { System.out.println("Hello, " + name); } greet(42); // 컴파일 에러: int를 String으로 바꿔주지 않음 // Python이었다면 실행 후 TypeError로 뒤늦게 발각 // var: 타입을 안 쓸 뿐, 여전히 정적 타입 var count = 10; // int로 추론 // count = "x"; // 컴파일 에러, count는 여전히 int var names = new ArrayList<String>(); // ArrayList<String>으로 추론 // var는 지역 변수에서만 허용된다 — 필드/매개변수/반환타입엔 못 씀

오토박싱 / 언박싱

int vs Integer, 겉보기엔 같아 보여도

제네릭은 primitive를 담지 못해서(List<int> 불가) 컴파일러가 자동으로 박싱/언박싱을 끼워 넣어요. 편리하지만 동등성 비교엔 함정이 있어요.

Boxing.java

int a = 10; Integer boxed = a; // 오토박싱: int → Integer int unboxed = boxed; // 오토언박싱: Integer → int Integer x = 100; Integer y = 100; System.out.println(x == y); // true — -128~127은 캐시된 같은 객체 Integer p = 200; Integer q = 200; System.out.println(p == q); // false — 캐시 범위 밖은 서로 다른 객체 System.out.println(p.equals(q)); // true — 값 비교는 항상 equals()로 Integer nullable = null; int crash = nullable; // NullPointerException! 언박싱은 .intValue() 호출과 같음

기본 문법 복습

배열 · 제어문 · 오버로딩, Python과 다른 점

문법 자체보다 "무엇이 Python과 다른가"에 집중해서 봐요.

Basics.java

// 배열: 크기가 고정, 타입도 고정 (Python list와 다름) int[] nums = {1, 2, 3}; nums[0] = 10; // nums.length는 메서드 호출이 아니라 필드 접근 (괄호 없음) System.out.println(nums.length); // switch: fall-through 주의, 요즘은 화살표 문법 권장 int day = 3; String label = switch (day) { case 1, 7 -> "주말"; default -> "평일"; }; // 메서드 오버로딩: 같은 이름, 다른 매개변수 시그니처 // Python엔 없는 개념 — 마지막 정의만 남는 Python과 달리 둘 다 공존한다 void log(String msg) { System.out.println(msg); } void log(String msg, int level) { System.out.println("[" + level + "] " + msg); } // 어떤 log()가 호출될지는 컴파일 타임에 인자 타입으로 결정됨

실습

hello-world 실행하고 바이트코드 들여다보기

01
프로젝트 생성 Maven: mvn archetype:generate, Gradle: gradle init --type java-application 으로 뼈대를 만든다.
02
Hello.java 작성 public class Hello { public static void main(String[] args) { System.out.println("Hello, Java"); } }
03
컴파일 & 실행 mvn compile exec:java 또는 gradle run (혹은 직접 javac Hello.java && java Hello)
04
javap로 바이트코드 확인 javap -c Hello.class 로 컴파일된 바이트코드 명령어(opcode)를 직접 눈으로 확인한다.
05
관찰하기 System.out.println 한 줄이 getstatic, invokevirtual 같은 여러 opcode로 풀려있는 걸 확인해보자.

흔한 실수

처음엔 다 이런 실수를 한다

01
== 로 객체 비교 Integer, String 같은 참조 타입을 ==로 비교하면 값이 아니라 주소를 비교한다. 값 비교는 항상 equals().
02
var를 아무 데나 쓰려는 시도 var는 지역 변수 전용. 필드, 매개변수, 반환 타입에는 쓸 수 없다.
03
IDE만 믿고 javac를 직접 안 돌려봄 IDE가 감춘 컴파일/클래스패스 문제를 javac/java 명령으로 직접 재현해보는 습관이 디버깅에 필수적이다.
04
GC가 즉시 회수한다는 착각 System.gc()는 "권고"일 뿐 강제가 아니다. Python의 즉시 참조 카운팅 회수와 달리 회수 시점을 보장하지 않는다.
05
의존성 버전 충돌을 손으로만 고치기 mvn dependency:tree, gradle dependencies 로 먼저 충돌 지점을 확인하고 고친다.

마무리

이번 주 배운 것, 셀프 체크

바이트코드가 뭔지, javac가 왜 필요한지 설명할 수 있나요?

소스코드를 그대로 실행하는 게 아니라 .class로 컴파일 후 JVM이 로드/해석/JIT 합니다.

CPython과 JVM의 GC 전략 차이를 설명할 수 있나요?

참조 카운팅 + 보조 GC vs 세대별 GC, 회수 시점 보장 여부가 다릅니다.

pom.xml과 build.gradle에서 의존성을 어떻게 선언하는지 기억하나요?

둘 다 groupId:artifactId:version 좌표로 Maven Central을 참조합니다.

Integer x == Integer y가 항상 값 비교가 아닌 이유는?

-128~127 캐시 범위 밖은 서로 다른 객체라서, 값 비교는 equals()를 써야 안전합니다.
