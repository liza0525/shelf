클래스 설계 · 생성자

생성자 오버로딩과 this() 체이닝

Python의 __init__은 하나뿐이라 기본값(default argument)으로 다양한 초기화를 처리한다. Java는 시그니처가 다른 생성자를 여러 개 두고, this()로 서로를 호출해 중복을 줄인다.

this() — 같은 클래스의 다른 생성자 호출

public class Book { private final String isbn; private final String title; private final String author; private boolean available; public Book(String isbn, String title, String author) { this(isbn, title, author, true); // 4개짜리 생성자로 위임 } public Book(String isbn, String title, String author, boolean available) { this.isbn = isbn; this.title = title; this.author = author; this.available = available; } }

super() — 부모 생성자 호출, 반드시 첫 줄

public class EBook extends Book { private final long fileSizeMb; public EBook(String isbn, String title, String author, long fileSizeMb) { super(isbn, title, author); // 항상 생성자의 첫 문장이어야 한다 this.fileSizeMb = fileSizeMb; } }

클래스 설계 · static vs instance

클래스가 갖는가, 인스턴스가 갖는가

static 멤버는 클래스 자체에 하나만 존재하고 모든 인스턴스가 공유한다. Python의 클래스 변수 및 @staticmethod/@classmethod와 개념은 비슷하지만, Java는 이를 명시적 키워드로 강제한다.

public class Book { private static int totalCount = 0; // 클래스 전체가 공유하는 정적 필드 private final String isbn; // 인스턴스마다 따로 갖는 필드 public Book(String isbn) { this.isbn = isbn; totalCount++; // 인스턴스 없이도 접근 가능 } public static int getTotalCount() { // static 메서드: this 없음, 인스턴스 필드 접근 불가 return totalCount; } } Book.getTotalCount(); // 인스턴스 생성 없이 클래스명으로 바로 호출

static 메서드 안에서는 this를 쓸 수 없다 — 애초에 어떤 인스턴스에도 속하지 않기 때문이다.

인터페이스 vs 추상 클래스

다중 구현이 필요하면 interface, 공통 상태가 필요하면 abstract class

interface

  • 필드는 사실상 public static final 상수뿐
  • 한 클래스가 여러 개를 동시에 implements 가능
  • default/static 메서드로 구현 제공(Java 8+)
  • Python의 Protocol(구조적 타이핑)에 가깝다

abstract class

  • 인스턴스 필드와 생성자를 가질 수 있다
  • 단일 상속만 가능 (extends 하나만)
  • 자식들과 공통 구현 로직을 공유
  • Python의 abc.ABC에 가깝다

interface · default/static 메서드

인터페이스도 구현을 가질 수 있다

public interface Lendable { boolean isAvailable(); void markAsLent(); // default: 구현체가 재정의하지 않아도 되는 기본 동작 default String lendingStatus() { return isAvailable() ? "대출 가능" : "대출 중"; } // static: 인터페이스 자체에 속하는 유틸리티 메서드 static Lendable emptySlot() { throw new UnsupportedOperationException("빈 슬롯은 대출 불가"); } }

Python ABC의 @abstractmethod는 Java의 추상 메서드와, ABC의 일반 메서드는 default 메서드와 대응된다. Protocol은 상속 없이 구조가 맞으면 인정되는 덕 타이핑이라 interface와는 강제성이 다르다.

상속 vs 컴포지션

합성 우선(Composition over Inheritance)

"코드를 재사용하고 싶다"는 이유만으로 extends를 쓰면 부모 클래스의 변경이 자식 전체로 번지는 fragile base class 문제가 생긴다. is-a가 아니라면 has-a로, 즉 필드로 조합하라.

상속 남용

  • is-a가 성립하지 않는데 억지로 extends
  • 부모 변경이 모든 자식에 파급된다
  • 단일 상속이라 조합의 유연성이 낮다
  • 부모의 private 구현에 자식이 암묵적으로 의존

컴포지션

  • has-a 관계를 필드로 명시적으로 표현
  • 인터페이스 조합으로 동작을 유연하게 조립
  • 각 컴포넌트를 독립적으로 교체/테스트
  • 다중 인터페이스 구현으로 다형성 확보

컴포지션 · final

상속 대신 조합, 그리고 상속 자체를 막기

public final class LoanRecord { // final class: 더 이상 상속 불가 private final Book book; // 상속 대신 필드로 소유 (has-a) private final Member borrower; private final LocalDate dueDate; public LoanRecord(Book book, Member borrower, LocalDate dueDate) { this.book = book; this.borrower = borrower; this.dueDate = dueDate; } public final boolean isOverdue() { // final method: 재정의 금지 return LocalDate.now().isAfter(dueDate); } }

final class는 String처럼 불변성·보안이 중요한 타입에, final method는 하위 클래스가 핵심 로직을 깨뜨리지 못하도록 막을 때 쓴다.

캡슐화 · 접근제어자

4단계 접근제어자로 노출 범위를 좁힌다

Python은 관례(_foo, __foo)로만 접근을 제한하고 강제성이 없다. Java는 컴파일러가 실제로 강제한다.

public

어디서나 접근 가능

클래스, 패키지와 무관하게 전부 노출된다. Python은 언더스코어 없는 이름이 관례상 이에 해당한다.

protected

같은 패키지 + 상속 자식

같은 패키지의 모든 클래스와, 다른 패키지라도 상속받은 자식 클래스에서 접근 가능. Python의 _foo 관례와 대응.

package-private

제어자 생략 시 기본값

같은 패키지 내에서만 접근 가능. 아무 키워드도 안 쓰면 이것이다. Python에는 정확히 대응하는 개념이 없다.

private

선언된 클래스 내부만

해당 클래스 안에서만 접근 가능. Python의 __foo(name mangling)와 비슷하지만 우회가 불가능하다.

equals · hashCode · toString

직접 오버라이드할 때 지켜야 할 계약

public class Book { private final String isbn; private final String title; @Override public boolean equals(Object o) { if (this == o) return true; if (!(o instanceof Book)) return false; Book other = (Book) o; return isbn.equals(other.isbn); // 동등성 기준을 명확히 정의 } @Override public int hashCode() { return Objects.hash(isbn); // equals에 쓰인 필드와 반드시 일치 } @Override public String toString() { return "Book{isbn=%s, title=%s}".formatted(isbn, title); } }

equals()로 같다고 판단되는 두 객체는 반드시 같은 hashCode()를 반환해야 한다 — HashMap/HashSet 동작의 전제 조건이다. Python의 __eq__/__hash__도 동일한 규칙을 따른다.

record · 보일러플레이트 제거

불변 데이터 클래스는 record 한 줄로

기존 방식 — 직접 구현한 보일러플레이트

public final class Isbn { private final String value; public Isbn(String value) { this.value = value; } public String value() { return value; } @Override public boolean equals(Object o) { if (this == o) return true; if (!(o instanceof Isbn)) return false; return value.equals(((Isbn) o).value); } @Override public int hashCode() { return Objects.hash(value); } @Override public String toString() { return "Isbn[value=" + value + "]"; } }

record — Java 17

public record Isbn(String value) {} // equals(), hashCode(), toString(), 접근자 value()까지 컴파일러가 자동 생성

record는 모든 필드가 final인 불변 데이터 홀더에만 적합하다. 가변 상태나 상속이 필요하면 여전히 일반 클래스가 필요하다. Python의 @dataclass(frozen=True)와 목적이 유사하다.

자주 하는 실수

이 패턴들을 피하면 절반은 성공

01
equals()만 오버라이드하고 hashCode()는 잊기 HashMap/HashSet에 넣으면 논리적으로 같은 객체인데도 찾지 못하는 버그가 생긴다.
02
equals/hashCode에 가변 필드 사용 HashSet에 넣은 뒤 그 필드를 바꾸면, 이후 같은 객체를 다시는 찾을 수 없게 된다.
03
"코드 재사용" 목적만으로 상속 사용 is-a 관계가 아닌데 extends부터 쓰면 합성 우선 원칙을 어기게 되고, 나중에 발목을 잡는다.
04
protected를 "자식 클래스 전용"으로 착각 실제로는 같은 패키지의 모든 클래스에도 열려 있다. 상속 자식뿐 아니라 패키지 전체가 접근할 수 있다.
05
인터페이스에 인스턴스 필드를 넣으려는 시도 인터페이스의 필드는 암묵적으로 public static final이라 인스턴스별 상태를 가질 수 없다.
