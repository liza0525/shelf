번외 · IDE 대안

Java는 IntelliJ IDEA로도 시작할 수 있어요

VSCode에 익숙하다면 그대로 써도 되지만, Java 생태계의 표준 IDE인 IntelliJ도 알아두면 도움이 됩니다. ← → 키로 페이지를 넘길 수 있어요.

왜 바꿔볼 만한가

VSCode의 Java 확장 vs IntelliJ 네이티브

VSCode의 Java 지원은 Red Hat 언어 서버가 뒤에서 별도 프로세스로 동작하는 방식이라, 대규모 리팩터링·복잡한 제네릭 추론에서 가끔 느려지거나 어긋날 때가 있어요. IntelliJ는 Java 전용으로 설계된 IDE라 이 부분이 훨씬 안정적입니다.

VSCode (지금까지 방식)

  • Extension Pack for Java 설치 필요
  • 큰 프로젝트에서 인덱싱·리팩터링이 느릴 수 있음
  • JAVA_HOME을 OS별로 직접 설정

IntelliJ IDEA (대안)

  • 설치 즉시 Java 전용 기능 내장, 확장 설치 불필요
  • 리팩터링/코드 탐색이 Java 특화로 더 정교함
  • IDE가 JDK를 자동으로 찾아주거나 다운로드해줌

준비물

설치할 건 딱 하나

VSCode 때와 달리 확장을 따로 찾아 설치할 필요가 없어요. IDE 자체가 Java 개발에 필요한 기능을 대부분 내장하고 있습니다.

01
IntelliJ IDEA Community Edition (무료) Ultimate는 유료(Spring/DB 도구 추가)지만, 이 커리큘럼 수준에서는 Community로 충분.
02
JDK는 설치 없이 IDE 안에서 바로 받기 프로젝트 생성 마법사에서 "Download JDK" 버튼으로 즉시 설치 가능 (OS 관계없이 동일한 과정).

IntelliJ 설치

OS별 설치 방법

JetBrains Toolbox App을 쓰면 OS 상관없이 설치·업데이트 방식이 동일해지지만, 각 OS의 직접 설치 방법도 함께 안내합니다.

Windows

  • Toolbox App 또는 공식 .exe installer
  • winget으로도 설치 가능
winget install JetBrains.IntelliJIDEA.Community

Linux

  • snap이 가장 간단
  • 또는 .tar.gz 압축 해제 후 bin/idea.sh 실행
sudo snap install intellij-idea-community --classic

macOS

  • Toolbox App 또는 brew cask
  • Apple Silicon 네이티브 빌드 지원
brew install --cask intellij-idea-ce

첫 실행

설치 직후, OS마다 한 번씩 걸리는 것

macOS

Gatekeeper 경고

App Store 밖에서 받은 앱이라 "확인되지 않은 개발자" 경고가 뜰 수 있음. Finder에서 우클릭 → 열기로 한 번만 허용하면 이후엔 정상 실행.

Windows

SmartScreen 경고

공식 사이트에서 받았다면 "추가 정보 → 실행"으로 진행. 이후 Windows Defender가 프로젝트 인덱싱을 느리게 만들 수 있어 뒤에서 예외 처리를 권장.

JDK 설정

JAVA_HOME을 직접 안 만져도 되는 이유

VSCode 버전 자료에서는 OS별로 JAVA_HOME 위치가 다 달랐죠. IntelliJ는 프로젝트 설정 안에서 JDK를 자체 관리하기 때문에, 이 과정이 OS 상관없이 완전히 동일합니다.

New Project 마법사 / File → Project Structure

SDK 드롭다운 → "Add SDK" → "Download JDK" // 벤더: Eclipse Temurin, 버전: 21 (LTS) 선택 → 자동 다운로드 및 등록 // Windows/Linux/macOS 모두 동일한 화면, 동일한 절차

프로젝트 생성

Maven 프로젝트 만들기

01
New Project → Maven Archetype 또는 그냥 New Project 언어: Java, 빌드 시스템: Maven 선택.
02
JDK 선택 (06번에서 등록한 21 선택) 없으면 그 자리에서 Download JDK로 즉시 추가.
03
pom.xml 자동 생성 확인 우측 Maven 툴윈도우에 프로젝트가 인식되면 정상.

키맵

Ctrl ↔ Cmd, 기본 키맵이 아예 갈린다

IntelliJ는 설치 시 "Windows/Linux" 키맵과 "macOS" 키맵 중 하나를 기본값으로 골라줘요. 같은 기능이라도 단축키 체계가 다르기 때문에, 다른 OS에서 쓰던 습관 그대로 오면 헷갈릴 수 있습니다.

Windows / Linux 키맵

Ctrl/Alt 중심

Search Everywhere는 Shift 두 번, 실행은 Shift+F10 계열.

macOS 키맵

Cmd/Option 중심

Windows/Linux의 Ctrl 자리를 Cmd가 대신하고, Alt 자리를 Option이 대신함. Preferences → Keymap에서 언제든 확인 가능.

VSCode 습관이 남아있다면 Plugins에서 "VSCode Keymap" 플러그인을 설치해 VSCode 단축키 그대로 쓸 수도 있어요.

디버거

launch.json 없이 바로 중단점

VSCode는 .vscode/launch.json 설정이 필요했지만, IntelliJ는 main 메서드 옆 실행 아이콘을 누르면 Run Configuration이 자동 생성돼요.

01
코드 라인 번호 왼쪽을 클릭해 중단점 설정 OS 관계없이 동일한 클릭 동작.
02
main 메서드 옆 초록 화살표 → Debug 선택 별도 설정 파일 없이 즉시 디버그 세션 시작.

자주 걸리는 문제

OS별 트러블슈팅

Win
인덱싱이 유난히 오래 걸림 Windows Defender 실시간 검사가 프로젝트 폴더를 스캔하면서 느려지는 경우가 많음. Defender 제외 목록에 프로젝트 폴더와 IntelliJ 캐시 폴더를 추가.
Linux
"Failed to add file watcher" 경고 기본 inotify 워치 개수 제한 때문. fs.inotify.max_user_watches 값을 늘려야 대형 프로젝트 인덱싱이 정상 동작함.
macOS
첫 실행이 "손상되었습니다"로 막힘 공증(notarization) 이슈나 다운로드 손상. brew cask로 재설치하거나 공식 사이트에서 다시 받으면 대부분 해결.

Linux: inotify 워치 한도 늘리기

echo fs.inotify.max_user_watches=524288 | sudo tee -a /etc/sysctl.conf sudo sysctl -p

마이그레이션 팁

VSCode 습관을 최대한 살리는 법

VSCode Keymap 플러그인

단축키 그대로 이식

Settings → Plugins에서 검색 설치. Ctrl+P(파일 검색) 등 익숙한 조합을 그대로 쓸 수 있음.

통합 터미널

기존 셸 습관 유지 가능

IntelliJ 하단 Terminal 탭도 VSCode처럼 프로젝트 루트에서 바로 열림. 셸 종류(bash/zsh/PowerShell)도 OS 기본값을 그대로 따름.

실습

첫 Java 프로그램, IntelliJ로 실행하기

01
New Project로 Maven 프로젝트 생성 07번 슬라이드 순서 그대로.
02
src/main/java에 App.java 작성 System.out.println("Hello, IntelliJ!")를 담은 main.
03
main 옆 초록 화살표로 Run 하단 Run 패널에 출력이 뜨면 성공.
04
중단점 찍고 Debug로 재실행 변수 값이 Debug 패널에 보이면 연동 성공.

마무리

VSCode냐 IntelliJ냐, 셀프 체크

IntelliJ의 JDK 자동 다운로드 기능을 써봤나요?

Project Structure → SDKs → Download JDK로 OS 상관없이 동일하게 진행돼요.

내 OS 기본 키맵(Windows/Linux vs macOS)이 뭔지 확인했나요?

Preferences → Keymap에서 현재 프리셋을 볼 수 있어요.

Run/Debug를 launch.json 없이 실행해봤나요?

main 메서드 옆 초록 화살표만 누르면 설정 파일이 자동 생성돼요.

두 IDE 중 어떤 걸 이 커리큘럼에서 계속 쓸지 정했나요?

VSCode도 충분히 좋은 선택이에요 — IntelliJ는 참고용 대안입니다.
