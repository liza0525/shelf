Framework

DI 컨테이너 위에서 계층 구조를 갖춘 서버 애플리케이션을 만듭니다.

Persistence

JPA / Hibernate

엔티티 클래스만으로 SQL 없이 DB와 연동합니다.

Capstone

board CRUD 재구현

Python으로 만들었던 그 게시판 API를 Java로 다시 만들고 테스트까지 붙입니다.

DEPENDENCY INJECTION

객체를 직접 만들지 않는다

Spring 컨테이너가 @Component 계열 애노테이션이 붙은 클래스를 인스턴스화하고, 필요한 곳에 생성자로 주입합니다. FastAPI의 Depends()와 비슷한 발상이지만 범위가 애플리케이션 전체입니다.

BOARDSERVICE.JAVA

// @Service: 이 클래스는 Spring이 관리하는 빈(Bean)이다 @Service public class BoardService { private final BoardRepository boardRepository; // 생성자가 하나뿐이면 @Autowired 생략 가능 (생성자 주입 권장 방식) public BoardService(BoardRepository boardRepository) { this.boardRepository = boardRepository; } public Board findById(Long id) { return boardRepository.findById(id) .orElseThrow(() -> new BoardNotFoundException(id)); } }

LAYERED ARCHITECTURE

Controller → Service → Repository

각 계층은 애노테이션으로 역할이 선언되고, 책임이 분리됩니다. Django의 view/serializer/model, FastAPI의 router/service/ORM 구조와 대응됩니다.

@RestController

HTTP 요청 처리

URL과 HTTP 메서드를 매핑하고, 요청/응답 바디를 JSON으로 자동 변환합니다.

@Service

비즈니스 로직

검증, 트랜잭션, 도메인 규칙을 담당하며 Controller와 Repository 사이를 중개합니다.

@Repository

데이터 접근

DB CRUD를 담당하는 계층. JPA에서는 인터페이스 선언만으로 구현체가 자동 생성됩니다.

JPA ENTITY

클래스가 곧 테이블이 된다

@Entity를 붙이면 Hibernate가 필드를 컬럼에 매핑합니다. Django ORM의 Model 클래스, SQLAlchemy의 declarative model과 개념이 거의 같습니다.

BOARD.JAVA

@Entity @Table(name = "boards") public class Board { @Id @GeneratedValue(strategy = GenerationType.IDENTITY) private Long id; @Column(nullable = false, length = 200) private String title; @Column(columnDefinition = "TEXT") private String content; @CreationTimestamp private LocalDateTime createdAt; protected Board() {} // JPA는 기본 생성자가 필요함 public Board(String title, String content) { this.title = title; this.content = content; } // getter만 공개, setter는 도메인 메서드로 대체 (필드 무분별 변경 방지) public Long getId() { return id; } public String getTitle() { return title; } }

REST ENDPOINTS

게시판 CRUD 엔드포인트 설계

JpaRepository를 상속하면 save/findById/findAll/deleteById가 공짜로 생깁니다. Controller는 이걸 조합해 HTTP 메서드에 매핑합니다.

GET /boards

목록 조회

boardRepository.findAll()로 전체 목록을 페이지네이션 없이 우선 반환합니다.

GET /boards/{id}

단건 조회

존재하지 않으면 BoardNotFoundException을 던져 404로 응답합니다.

POST /boards

생성

@RequestBody로 JSON을 DTO로 받아 @Valid로 먼저 검증합니다.

PUT / DELETE /boards/{id}

수정 · 삭제

수정은 영속 상태 엔티티의 필드를 바꿔 더티 체킹으로 자동 UPDATE됩니다.

CONTROLLER

@RestController로 라우팅하기

클래스 레벨 @RequestMapping으로 공통 경로를 잡고, 메서드마다 @GetMapping/@PostMapping 등으로 세부 경로를 붙입니다.

BOARDCONTROLLER.JAVA

@RestController @RequestMapping("/boards") public class BoardController { private final BoardService boardService; public BoardController(BoardService boardService) { this.boardService = boardService; } @GetMapping public List<BoardResponse> list() { return boardService.findAll(); } @PostMapping @ResponseStatus(HttpStatus.CREATED) public BoardResponse create(@Valid @RequestBody BoardCreateRequest request) { return boardService.create(request); } @DeleteMapping("/{id}") @ResponseStatus(HttpStatus.NO_CONTENT) public void delete(@PathVariable Long id) { boardService.delete(id); } }

EXCEPTION HANDLING

계층마다 throw, 한 곳에서 catch

Service에서는 도메인 예외만 던지고, 그걸 HTTP 응답으로 바꾸는 책임은 @RestControllerAdvice 한 곳에 모읍니다. FastAPI의 exception_handler와 같은 패턴입니다.

GLOBALEXCEPTIONHANDLER.JAVA

@RestControllerAdvice public class GlobalExceptionHandler { @ExceptionHandler(BoardNotFoundException.class) public ResponseEntity<ErrorResponse> handleNotFound(BoardNotFoundException e) { ErrorResponse body = new ErrorResponse("BOARD_NOT_FOUND", e.getMessage()); return ResponseEntity.status(HttpStatus.NOT_FOUND).body(body); } @ExceptionHandler(MethodArgumentNotValidException.class) public ResponseEntity<ErrorResponse> handleInvalid(MethodArgumentNotValidException e) { // @Valid 검증 실패 시 스프링이 자동으로 던지는 예외 String message = e.getBindingResult().getFieldError().getDefaultMessage(); return ResponseEntity.badRequest().body(new ErrorResponse("INVALID_INPUT", message)); } }

VALIDATION

DTO에 제약 조건을 선언한다

bean-validation 애노테이션을 DTO 필드에 붙이고 컨트롤러 파라미터에 @Valid만 추가하면 검증이 요청 처리 이전에 자동으로 실행됩니다. Pydantic의 Field 제약과 동일한 발상입니다.

BOARDCREATEREQUEST.JAVA

public record BoardCreateRequest( @NotBlank(message = "제목은 비어 있을 수 없습니다") @Size(max = 200, message = "제목은 200자를 넘을 수 없습니다") String title, @NotBlank(message = "내용은 비어 있을 수 없습니다") String content ) {} // 컨트롤러에서는 이렇게만 쓰면 검증이 먼저 실행됨 // public BoardResponse create(@Valid @RequestBody BoardCreateRequest request)

VS PYTHON

FastAPI/Django 대비 무엇이 다른가

개념은 대부분 대응되지만, Java 쪽은 컴파일 타임에 더 많은 걸 강제합니다.

Python (FastAPI/Django)

  • Depends()로 함수 단위 의존성 주입
  • 데코레이터 기반 라우팅, 런타임에 등록
  • 타입힌트는 선택, 런타임 강제 아님
  • Pydantic으로 요청 검증

Java (Spring Boot)

  • IoC 컨테이너가 빈 생명주기 전체를 관리
  • 애노테이션 기반 라우팅, 컴파일 타임 타입 체크
  • 생성자 시그니처가 곧 의존성 계약
  • bean-validation으로 요청 검증

TESTING

JUnit 5 + Mockito로 단위 테스트

Repository는 @Mock으로 가짜 객체를 만들고, Service만 실제로 실행해 비즈니스 로직만 검증합니다. pytest + unittest.mock 조합과 동일한 목적입니다.

BOARDSERVICETEST.JAVA

@ExtendWith(MockitoExtension.class) class BoardServiceTest { @Mock private BoardRepository boardRepository; @InjectMocks private BoardService boardService; @Test void 존재하지_않는_게시글을_조회하면_예외를_던진다() { // given given(boardRepository.findById(1L)).willReturn(Optional.empty()); // when & then assertThrows(BoardNotFoundException.class, () -> boardService.findById(1L)); } }
