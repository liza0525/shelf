GENERICS BASICS

제네릭 기초: <T>와 타입 경계

Python의 TypeVar는 힌트일 뿐 런타임에 강제되지 않지만, Java 제네릭은 컴파일 타임에 타입을 검사합니다.

// 제네릭 클래스 - 타입 매개변수 T class Box<T> { private T value; public Box(T value) { this.value = value; } public T get() { return value; } } Box<String> box = new Box<>("hello"); // 타입 안전, 캐스팅 불필요 // 타입 경계(bounded type) - extends // T는 Number이거나 그 하위 타입이어야 함 static <T extends Number> double sum(List<T> list) { double total = 0; for (T n : list) total += n.doubleValue(); return total; } // Python typing.TypeVar는 힌트만 제공, 런타임 강제 없음 // Java는 컴파일 타임에 타입 불일치를 잡아냄

WILDCARDS

와일드카드: ? extends, ? super

PECS 원칙 — Producer-Extends, Consumer-Super. "꺼내기만 하면 extends, 넣기만 하면 super."

// ? extends T : 읽기 전용(Producer) - "생산자" static double sumAll(List<? extends Number> list) { double total = 0; for (Number n : list) total += n.doubleValue(); return total; // list.add(1) 은 컴파일 에러! } // ? super T : 쓰기 전용(Consumer) - "소비자" static void addIntegers(List<? super Integer> list) { list.add(1); list.add(2); // 읽을 때는 Object로만 꺼낼 수 있음 } sumAll(List.of(1, 2, 3)); // List<Integer> OK addIntegers(new ArrayList<Number>()); // List<Number> OK

LIST IMPLEMENTATIONS

List 구현체 비교

둘 다 List<T> 인터페이스를 구현하지만 내부 자료구조가 다릅니다.

Array-based

ArrayList

내부적으로 Object[] 배열 기반.
get(index): O(1)
끝에 add: O(1) (상각)
중간 삽입/삭제: O(n) — 요소 shift
캐시 지역성이 좋아 순회가 빠름. Python의 list와 구조가 거의 동일.

Node-based

LinkedList

이중 연결 리스트(Doubly Linked List).
get(index): O(n)
양 끝 add/remove: O(1) — Deque 구현체
중간 삽입/삭제: 위치를 이미 알면 O(1), 탐색 포함 시 O(n)
Python 표준 타입 중엔 없고 collections.deque가 가장 유사.

SET IMPLEMENTATIONS

Set 구현체 비교

중복을 허용하지 않는다는 점은 같지만, 순서 보장 여부와 성능이 다릅니다.

Hash Table

HashSet

해시테이블 기반. 순서 보장 안 함.
add/contains: 평균 O(1)
가장 빠르지만 순서가 필요 없을 때만 사용.

Red-Black Tree

TreeSet

정렬 트리 기반. 정렬된 순서 유지.
add/contains: O(log n)
Comparable/Comparator 필요.

Hash + Linked List

LinkedHashSet

HashSet + 연결리스트로 삽입 순서 유지.
add/contains: 평균 O(1)
빠른 조회 + 순서 보존이 모두 필요할 때.

MAP IMPLEMENTATIONS

Map 구현체 비교

가장 헷갈리는 부분: 어떤 Map이 순서를 보장하는가?

순서 보장 X

HashMap

해시테이블(버킷) 기반.
get/put: 평균 O(1)
Python dict와 비슷해 보이지만 순서를 전혀 보장하지 않음.

Key 정렬 순서

TreeMap

Red-Black Tree 기반.
get/put: O(log n)
key 기준 정렬 유지. firstKey(), ceilingKey() 등 NavigableMap API.

삽입/접근 순서

LinkedHashMap

HashMap + 이중연결리스트.
get/put: 평균 O(1)
삽입 순서 유지, accessOrder=true로 LRU 캐시 구현 가능.

INTERNAL STRUCTURE

Python dict vs Java HashMap

겉보기엔 비슷한 키-값 자료구조지만 내부 설계 철학이 다릅니다.

Python dict

  • CPython 3.7+: 삽입순서 배열 + 해시 인덱스 배열의 "컴팩트 dict"
  • 삽입 순서 보장이 언어 스펙에 명시됨
  • 리사이즈 시 전체 재해시, 초기 크기 8부터 확장
  • GIL 덕분에 사실상 순차 접근, 별도 동시성 고려 적음

Java HashMap

  • 버킷 배열 + 각 버킷은 연결리스트(또는 트리, 8개 초과 시)
  • 순서 보장 안 함 — 필요하면 LinkedHashMap을 명시적으로 사용
  • load factor 0.75 초과 시 버킷 배열 2배로 확장
  • 멀티스레드 환경에서는 별도로 ConcurrentHashMap 사용 필요

NULL SAFETY

Optional<T>로 null 안전성 다루기

Optional은 "null일 수도 있다"를 타입 시스템으로 드러내는 도구입니다.

// 안 좋은 패턴: Optional.get() 남발 // NoSuchElementException 위험 — null 체크와 다를 바 없음 User user = findUser(id).get(); // 권장 1: orElse / orElseGet User user = findUser(id).orElse(User.GUEST); User user = findUser(id).orElseGet(() -> createDefaultUser()); // 권장 2: orElseThrow로 의도를 명시 User user = findUser(id) .orElseThrow(() -> new UserNotFoundException(id)); // 권장 3: map/filter로 체이닝 (Stream과 유사) String name = findUser(id) .map(User::getName) .filter(n -> !n.isBlank()) .orElse("이름 없음"); // 권장 4: ifPresentOrElse findUser(id).ifPresentOrElse( u -> System.out.println("찾음: " + u), () -> System.out.println("없음") );

CONVENTION COMPARE

Optional vs Python None 체크

둘 다 "값이 없을 수 있음"을 다루지만 강제력의 정도가 다릅니다.

Python None 체크

  • if user is None: ... — 관행적 체크, 강제성 없음
  • 타입힌트 Optional[User]를 써도 무시하고 접근 가능
  • 깜빡하면 AttributeError는 런타임에만 발견
  • 정적 분석 도구(mypy 등)에 의존해야 사전에 잡힘

Java Optional<T>

  • 반환 타입 자체가 Optional<User> — 시그니처에 "없을 수도 있음"이 드러남
  • 컨벤션상 필드/파라미터엔 쓰지 않고 반환값에만 사용
  • get() 대신 orElseThrow 등으로 처리하지 않으면 코드 리뷰에서 지적받는 문화
  • 컴파일러가 강제하진 않지만 팀 컨벤션으로 안전장치를 만듦

HANDS-ON PREVIEW

실습: CSV → Map<String, List<T>> 그룹핑

부서(dept), 이름(name) 두 컬럼의 CSV를 읽어 부서별 이름 목록으로 그룹핑하는 유틸리티를 작성합니다.

// 목표: CSV(부서,이름)를 읽어 부서별 이름 리스트로 그룹핑 public static Map<String, List<String>> groupByDept(Path csvPath) throws IOException { Map<String, List<String>> result = new LinkedHashMap<>(); // 입력 순서 유지 try (BufferedReader reader = Files.newBufferedReader(csvPath)) { String line; while ((line = reader.readLine()) != null) { String[] cols = line.split(","); String dept = cols[0].trim(); String name = cols[1].trim(); result.computeIfAbsent(dept, k -> new ArrayList<>()) .add(name); } } return result; } // Python이라면: // from collections import defaultdict // groups = defaultdict(list) // for dept, name in rows: groups[dept].append(name)

COMMON PITFALLS

흔한 실수

제네릭과 컬렉션을 다룰 때 자주 걸려 넘어지는 지점들입니다.

01

Optional.get() 남발 null 체크를 Optional.get()으로만 바꾼 것은 개선이 아닙니다. orElseThrow / map으로 의도를 드러내세요.

02

와일드카드 방향 헷갈림 (PECS) 꺼내기만 하면 extends, 넣기만 하면 super. 반대로 쓰면 컴파일 에러가 납니다.

03

HashMap 순서에 의존한 로직 우연히 순서가 유지되는 것처럼 보여도 스펙상 보장되지 않습니다. 필요하면 LinkedHashMap/TreeMap을 명시하세요.

04

equals/hashCode 없는 객체를 Hash 계열 키로 사용 HashSet/HashMap에서 중복 제거와 조회가 제대로 동작하지 않습니다.

05

Optional을 필드/파라미터 타입으로 사용 관례상 반환 타입에만 사용합니다. 필드에 쓰면 직렬화 문제와 불필요한 래핑이 생깁니다.
