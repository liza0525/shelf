std::thread::spawn

클로저 하나를 새 OS 스레드에서 실행합니다. join()을 호출하기 전까지 메인 스레드는 계속 진행돼요.

spawn과 join

use std::thread; fn main() { // 새 스레드에서 클로저를 실행 let handle = thread::spawn(|| { for i in 1..5 { println!("스레드에서: {i}"); } }); for i in 1..3 { println!("메인에서: {i}"); } // 스레드가 끝날 때까지 대기 (Result<T, E> 반환) handle.join().unwrap(); }

6주차 · 동시성

GIL이 없다면

Python의 스레드는 CPU 코어를 동시에 못 씁니다. Rust의 스레드는 진짜 병렬로 돌아갑니다.

Python

  • GIL(Global Interpreter Lock) 때문에 한 번에 바이트코드 1개만 실행
  • 스레드를 여러 개 만들어도 CPU-bound 코드는 병렬 아님
  • 진짜 병렬 처리를 하려면 multiprocessing 필요
  • 데이터 레이스는 런타임에만 드러남

Rust

  • 스레드 하나 = 진짜 OS 스레드, 코어마다 병렬 실행
  • 전역 락(GIL) 자체가 없음
  • Send / Sync 트레잇으로 컴파일러가 안전성을 검사
  • 데이터 레이스는 대부분 컴파일 단계에서 막힘

move 클로저로 소유권 넘기기

새 스레드는 메인 스레드보다 오래 살 수도 있어서, 빌려온 참조 대신 값을 통째로 넘겨받아야 안전합니다.

move 키워드

use std::thread; fn main() { let data = vec![1, 2, 3]; // move: data의 소유권을 클로저 안으로 이동 let handle = thread::spawn(move || { println!("스레드가 소유한 데이터: {:?}", data); }); // data는 이미 이동했으므로 여기서 다시 쓰면 컴파일 에러 // println!("{:?}", data); handle.join().unwrap(); }

6주차 · 동시성

컴파일이 안 되는 게 기능입니다

공유 변수를 락 없이 여러 스레드가 건드리면, Rust는 실행조차 시켜주지 않습니다.

컴파일 에러 예시

use std::thread; fn main() { let mut counter = 0; let mut handles = vec![]; for _ in 0..10 { let handle = thread::spawn(|| { counter += 1; // ← 에러 }); handles.push(handle); } // error[E0373]: closure may outlive the current function, // but it borrows `counter`, which is owned by the current function // help: to force the closure to take ownership ... use `move` }

Python/Java였다면 일단 실행은 되고, 운이 나쁠 때만 레이스 컨디션이 터집니다. Rust는 그 가능성 자체를 컴파일 단계에서 차단해요.

Arc<T>로 함께 소유하기

Arc = Atomic Reference Counted. Python의 참조 카운팅과 비슷하지만, 카운트 증가/감소가 원자적(atomic)이라 스레드 간에 안전합니다.

Arc::clone으로 참조 공유

use std::sync::Arc; use std::thread; fn main() { let data = Arc::new(vec![1, 2, 3]); let mut handles = vec![]; for i in 0..3 { let data = Arc::clone(&data); // 값 복사 아님, 참조 카운트만 +1 handles.push(thread::spawn(move || { println!("스레드 {i}: {:?}", data); })); } for h in handles { h.join().unwrap(); } }

6주차 · 동시성

Arc<Mutex<T>> 조합

Arc는 "누가 소유하는가"를, Mutex는 "누가 접근하는가"를 책임집니다. 둘을 합치면 여러 스레드가 안전하게 값을 바꿀 수 있어요.

lock()으로 상호 배제 접근

use std::sync::{Arc, Mutex}; use std::thread; fn main() { let counter = Arc::new(Mutex::new(0)); let mut handles = vec![]; for _ in 0..10 { let counter = Arc::clone(&counter); handles.push(thread::spawn(move || { let mut num = counter.lock().unwrap(); // 잠금 획득 *num += 1; })); // 스코프를 벗어나면 자동으로 잠금 해제 } for h in handles { h.join().unwrap(); } println!("결과: {}", *counter.lock().unwrap()); // 10 }

mpsc 채널로 메시지 주고받기

mpsc = multiple producer, single consumer. "메모리를 공유해서 통신하지 말고, 통신해서 메모리를 공유하라"는 Go의 철학과 닮았습니다.

tx.send / rx.recv

use std::sync::mpsc; use std::thread; fn main() { let (tx, rx) = mpsc::channel(); for i in 0..3 { let tx = tx.clone(); // producer는 여러 개 만들 수 있음 thread::spawn(move || { tx.send(format!("작업 {i} 완료")).unwrap(); }); } drop(tx); // 원본 tx를 닫아야 아래 for 루프가 끝남 for msg in rx { // consumer는 하나뿐 println!("{msg}"); } }

Go의 channel, Python의 queue.Queue와 개념은 비슷하지만, 타입과 소유권이 컴파일 타임에 검사된다는 점이 다릅니다.

async/await, 왜 필요할까

스레드는 강력하지만 비쌉니다. I/O 대기가 많은 작업(웹 서버 등)에서는 훨씬 가벼운 동시성 모델이 필요해요. 지금은 개념만, 8주차 axum 서버에서 실전 사용합니다.

Thread

OS가 스케줄링

스레드마다 스택 메모리를 따로 할당. 수천 개를 띄우면 부담이 큽니다.

Async

런타임(tokio)이 스케줄링

.await 지점에서만 다른 작업으로 전환. 훨씬 가볍게 수만 개를 동시에 다룰 수 있습니다.

맛보기 (문법만)

async fn fetch_data() -> String { // .await 지점에서 다른 작업으로 전환될 수 있음 "데이터".to_string() } #[tokio::main] async fn main() { let result = fetch_data().await; println!("{result}"); }

흔한 실수

01

데드락(deadlock) 두 개의 Mutex를 스레드마다 다른 순서로 lock하면 서로 상대방의 잠금 해제를 기다리며 영원히 멈춥니다.

02

Arc 없이 데이터를 여러 스레드에 넘기려는 시도 move로 값을 넘기면 소유권이 첫 스레드로 이동해버려서, 다음 반복에서 또 쓰려고 하면 "value used after move" 컴파일 에러가 납니다.

03

MutexGuard를 너무 오래 들고 있기 lock()의 반환값을 변수에 담아 오래 살려두면, 다른 스레드가 그동안 계속 대기합니다. 락은 최소한의 스코프에서만 잡으세요.

04

async fn을 호출만 하고 .await를 안 붙임 async 함수 호출은 그 자체로 실행되지 않고 Future 값만 만듭니다. .await가 없으면 컴파일러 경고만 뜨고 아무 일도 일어나지 않아요.
