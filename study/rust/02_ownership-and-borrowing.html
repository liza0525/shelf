Ownership · 소유권

GC 없이 메모리를 관리하는 방법

Rust는 가비지 컬렉터도, 수동 free()도 없다. 대신 컴파일러가 검사하는 세 가지 규칙으로 메모리를 관리한다.

규칙 1

하나의 owner

모든 값(value)은 그 값을 소유하는 변수, 즉 owner를 단 하나만 가진다.

규칙 2

스코프를 벗어나면 drop

owner가 스코프(scope)를 벗어나는 순간, 값은 자동으로 drop되어 메모리가 즉시 해제된다.

규칙 3

동시에 owner는 하나

값을 다른 변수에 대입하면 소유권이 그대로 이동(move)한다. 복사가 아니라 이동이다.

Ownership · Move Semantics

언어마다 메모리를 관리하는 방식이 다르다

Python과 Java는 런타임이 알아서 치워주고, C는 직접 치워야 한다. Rust는 그 중간, 컴파일타임에 규칙을 강제한다.

Python

참조 카운팅 + GC

객체마다 참조 개수를 추적하다가 0이 되면 회수한다. 순환 참조는 별도의 GC가 처리한다. 개발자는 메모리를 신경 쓸 필요가 거의 없다.

Java

가비지 컬렉터(GC)

더 이상 참조되지 않는 객체를 GC 스레드가 주기적으로 찾아 회수한다. 편리하지만 GC 실행 시점에 예측하기 어려운 지연이 생길 수 있다.

C

수동 malloc / free

개발자가 직접 메모리를 할당하고 해제한다. 자유롭지만 free를 잊으면 메모리 누수, 두 번 free하면 이중 해제 버그가 생긴다.

Rust

컴파일타임 소유권 검사

런타임 GC도, 수동 free도 없다. 컴파일러가 소유권 규칙을 정적으로 검사해서, 규칙을 어기는 코드는 애초에 컴파일되지 않는다.

Ownership · Move

대입하면 소유권이 이동한다

❌ 컴파일 에러가 나는 코드

// String은 힙에 데이터를 저장하는 타입 fn main() { let s1 = String::from("hello"); let s2 = s1; // s1의 소유권이 s2로 이동(move) println!("{}, world!", s1); // error[E0382]! }

✅ 고친 코드 — clone으로 깊은 복사

fn main() { let s1 = String::from("hello"); let s2 = s1.clone(); // 힙 데이터를 통째로 복제 println!("{}, world!", s1); // OK, s1은 여전히 유효하다 }

Ownership · Copy vs Move

모든 대입이 소유권을 이동시키진 않는다

스택에만 저장되는 고정 크기 타입은 Copy 트레잇을 구현해서, 대입할 때 소유권 이동 대신 값 복사가 일어난다.

Move — 힙 데이터

  • String, Vec<T>, Box<T> 등
  • 대입하면 소유권이 이동한다
  • 원본 변수는 이동 후 사용할 수 없다
  • 이동 후 원본을 쓰면 컴파일 에러

Copy — 스택 데이터

  • i32, f64, bool, char 등
  • 대입하면 값이 그대로 복사된다
  • 원본과 복사본을 모두 계속 쓸 수 있다
  • Copy 타입끼리의 튜플도 Copy

Borrowing · 빌림

소유권을 넘기지 않고 값을 빌려준다

& — 불변 참조 (immutable reference)

fn main() { let s = String::from("hello"); let len = calculate_length(&s); // s를 "빌려준다" println!("'{}'의 길이는 {}", s, len); // s를 계속 쓸 수 있다 } fn calculate_length(s: &String) -> usize { s.len() } // s는 참조일 뿐이라 여기서 값이 drop되지 않는다

&mut — 가변 참조 (mutable reference)

fn main() { let mut s = String::from("hello"); change(&mut s); println!("{}", s); // "hello, world" } fn change(s: &mut String) { s.push_str(", world"); }

Borrowing · 빌림 규칙

같은 스코프에서, 딱 하나의 형태만

여러 개의 불변 참조는 동시에 가능하지만, 가변 참조는 단 하나뿐이며 불변 참조와도 함께 존재할 수 없다.

OK

불변 참조는 여러 개

let r1 = &s; let r2 = &s; — 아무도 값을 바꾸지 않으므로 동시에 여러 개를 가져도 안전하다.

OK

가변 참조는 하나만

let r1 = &mut s; — 한 스코프에서 &mut는 동시에 딱 하나까지만 만들 수 있다.

NO

혼합은 금지

&mut가 살아있는 동안에는 &도 만들 수 없다. 이 규칙이 데이터 레이스를 컴파일타임에 막는다.

Borrowing · 컴파일 에러

규칙을 어기면 컴파일이 되지 않는다

❌ 가변 참조를 두 번 만든 코드

fn main() { let mut s = String::from("hello"); let r1 = &mut s; let r2 = &mut s; // error! println!("{}, {}", r1, r2); }

에러 메시지

error[E0499]: cannot borrow `s` as mutable more than once at a time --> src/main.rs:5:14 | 4 | let r1 = &mut s; | ------ first mutable borrow occurs here 5 | let r2 = &mut s; | ^^^^^^ second mutable borrow occurs here 6 | println!("{}, {}", r1, r2); | -- first borrow later used here

Lifetime · 댕글링 참조

참조는 값보다 오래 살아남을 수 없다

❌ 댕글링 참조를 반환하는 코드

fn dangle() -> &String { // error! let s = String::from("hello"); &s // s를 가리키는 참조를 반환하려 한다 } // 여기서 s는 스코프를 벗어나 drop된다

에러 메시지

error[E0106]: missing lifetime specifier --> src/main.rs:1:16 | 1 | fn dangle() -> &String { | ^ expected named lifetime parameter | = help: this function's return type contains a borrowed value, but there is no value for it to be borrowed from

Rust는 댕글링 참조가 생길 가능성이 있는 코드는 애초에 컴파일하지 않는다.

Lifetime · 'a 표기법

컴파일러에게 생존 기간을 알려준다

'a는 실제 값이 아니다. "이 참조들은 최소한 같은 기간 동안 유효하다"는 사실을 컴파일러에게 알려주는 표시일 뿐이다.

가장 긴 문자열을 반환하는 함수

fn longest<'a>(x: &'a str, y: &'a str) -> &'a str { if x.len() > y.len() { x } else { y } } fn main() { let s1 = String::from("long string"); let s2 = String::from("short"); let result = longest(s1.as_str(), s2.as_str()); println!("가장 긴 문자열: {}", result); }

x, y, 반환값이 모두 같은 'a 동안 유효해야 한다는 제약을 표현할 뿐, 값 자체를 바꾸지는 않는다.

String vs &str

Python에는 없는, owned vs borrowed 문자열

String — 소유한 문자열

  • 힙에 저장되고 소유권을 가진다
  • push_str 등으로 크기가 늘어날 수 있다
  • 다른 곳에 넘기려면 move 또는 clone

&str — 빌려온 문자열 슬라이스

  • 소유권이 없는 문자열 view
  • 문자열 리터럴 "hello"는 기본이 &str
  • 함수 인자로 받으면 더 유연하다

함수 매개변수는 &str이 더 유연하다

fn greet(name: &str) { // &str이면 둘 다 받을 수 있다 println!("안녕, {}!", name); } let owned = String::from("Rust"); // String let literal = "Python"; // &str greet(&owned); // String -> &str (deref coercion) greet(literal); // &str 그대로 전달

자주 만나는 컴파일 에러

이 에러들과 친해지면 편해진다

01
cannot borrow `s` as mutable more than once at a time 같은 스코프에서 &mut 참조를 두 개 이상 만들었을 때
02
cannot borrow `s` as immutable because it is also borrowed as mutable &mut 참조가 살아있는 동안 &참조를 추가로 만들었을 때
03
borrow of moved value: `s1` 소유권이 이동(move)된 변수를 이동 이후에 다시 사용했을 때
04
missing lifetime specifier 참조를 반환하는 함수인데 lifetime이 없어 컴파일러가 유효 기간을 추론하지 못할 때
05
cannot move out of `*s` which is behind a shared reference 참조(&T) 뒤에 있는 값을 소유권째로 꺼내려 했을 때

마무리

런타임 비용 없이 안전을 얻는다

소유권, 빌림, 라이프타임은 이후 모든 Rust 코드의 기반이 된다. 다음 장에서는 struct와 enum으로 데이터를 표현하는 법을 배운다.

Ownership

값마다 owner는 하나

스코프를 벗어나면 자동 drop, 대입은 기본적으로 move.

Borrowing

& / &mut

불변 참조는 여러 개, 가변 참조는 스코프당 하나만.

Lifetime

'a 힌트

참조는 가리키는 값보다 오래 살 수 없다는 규칙을 컴파일러에게 알려준다.
