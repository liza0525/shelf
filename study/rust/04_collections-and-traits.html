Vec<T>: 동적 배열

Python의 list처럼 크기가 자동으로 늘어나는 배열입니다. 다만 하나의 Vec 안에는 같은 타입 T만 들어갈 수 있어요.

Python list

  • 타입이 섞여도 됨 [1, "a", 3.0]
  • 런타임에 타입 체크
  • append / pop / 인덱싱

Rust Vec<T>

  • 모든 원소가 동일한 타입 T
  • 컴파일 타임에 타입 확정
  • push / pop / 인덱싱 (범위 벗어나면 panic)

Vec 기본 사용

let mut nums: Vec<i32> = Vec::new(); nums.push(10); nums.push(20); nums.push(30); let last = nums.pop(); // Some(30), Vec은 [10, 20] println!("{}", nums[0]); // 인덱싱, 범위 밖이면 panic! // 매크로로 바로 생성 let nums2 = vec![1, 2, 3]; // Python: nums = []; nums.append(10); nums.pop()

Vec 안의 값도 소유권을 따른다

Vec을 순회하는 방법에 따라 값이 이동(move)되는지, 빌려오는(borrow)지가 달라집니다.

순회 방식과 소유권

let strings = vec![String::from("a"), String::from("b")]; for s in strings { // into_iter(): 소유권이 이동 println!("{}", s); } // println!("{:?}", strings); // ❌ 이미 moved, 컴파일 에러 let strings2 = vec![String::from("a"), String::from("b")]; for s in &strings2 { // iter(): 참조로 순회 (빌림) println!("{}", s); } println!("{:?}", strings2); // ✅ 여전히 사용 가능

Python은 list를 순회해도 원본이 그대로 있는 게 당연하지만, Rust는 "누가 값을 소유하는가"를 항상 명시적으로 따집니다.

HashMap<K, V>: 키-값 저장소

Python dict

  • d = {}; d["a"] = 1
  • d.get("a") → None 가능
  • d.setdefault(k, v)

Rust HashMap<K, V>

  • use std::collections::HashMap
  • get()은 Option<&V> 반환
  • entry() API로 조건부 삽입

insert · get · entry

use std::collections::HashMap; let mut scores: HashMap<String, i32> = HashMap::new(); scores.insert(String::from("alice"), 90); scores.insert(String::from("bob"), 85); match scores.get("alice") { Some(v) => println!("alice: {v}"), None => println!("no entry"), } // entry: 키가 없을 때만 기본값 삽입 scores.entry(String::from("carol")).or_insert(0); // 있으면 값을 꺼내 수정, 없으면 0으로 시작해서 수정 *scores.entry(String::from("alice")).or_insert(0) += 1;

Iterator: map · filter · collect

결과만 보면 Python의 list comprehension과 비슷하지만, Rust의 iterator adaptor는 lazy합니다. collect나 for처럼 "소비"하는 시점이 와야 실제로 실행됩니다.

체이닝 예제

let nums = vec![1, 2, 3, 4, 5, 6]; let result: Vec<i32> = nums .iter() .filter(|&&n| n % 2 == 0) // 짝수만 남김 (아직 실행 안 됨) .map(|n| n * n) // 제곱 (아직 실행 안 됨) .collect(); // 여기서 비로소 실행! println!("{:?}", result); // [4, 16, 36] // Python: [n**2 for n in nums if n % 2 == 0]

타입도 강하게 개입합니다: collect()가 무엇으로 모을지는 좌변의 타입 표기(Vec<i32>)로 결정됩니다.

Trait: Java interface에 대응

trait은 "이 타입이 어떤 메서드를 구현해야 하는가"를 정의합니다. Java의 interface와 목적은 같지만 문법과 세부 동작이 다릅니다.

Java interface

  • interface Greet { String greet(); }
  • class가 implements로 구현
  • default 키워드로 기본 구현

Rust trait

  • trait Greet { fn greet(&self) -> String; }
  • struct가 impl Trait for Type으로 구현
  • trait 안에 본문을 두면 그게 default method

5주차 · 컬렉션과 트레잇

trait 정의 · impl · default method

Greet 트레잇

trait Greet { fn name(&self) -> String; // 구현 필수 (본문 없음) fn greet(&self) -> String { // default method format!("안녕하세요, {}입니다", self.name()) } } struct Person { name: String, } impl Greet for Person { fn name(&self) -> String { self.name.clone() } // greet()은 구현하지 않아도 default가 그대로 쓰임 } let p = Person { name: String::from("지영") }; println!("{}", p.greet()); // "안녕하세요, 지영입니다"

Generic: 함수와 구조체에 타입 매개변수

Java의 <T>와 비슷한 개념이지만, Rust는 트레잇 바운드를 함께 명시해서 "T가 무엇을 할 수 있는지"까지 컴파일 타임에 검사합니다.

제네릭 함수 · 구조체

fn largest<T: PartialOrd>(list: &[T]) -> &T { let mut max = &list[0]; for item in list { if item > max { max = item; } } max } struct Pair<T> { first: T, second: T, } impl<T: std::fmt::Display> Pair<T> { fn show(&self) { println!("({}, {})", self.first, self.second); } } let p = Pair { first: 1, second: 2 }; p.show(); // (1, 2)

5주차 · 컬렉션과 트레잇

impl Trait: 트레잇 바운드를 간단히

"이 타입은 몰라도 되고, 이 trait만 구현하면 된다"를 짧게 쓰는 문법입니다. 매개변수와 반환 타입 모두에 쓸 수 있어요.

impl Trait 문법

// 매개변수: T: IntoIterator<Item = i32> 를 줄여 씀 fn print_all(items: impl IntoIterator<Item = i32>) { for i in items { println!("{i}"); } } // 반환 타입에도 사용 가능 (구체 타입은 감춰짐) fn make_greeter(name: String) -> impl Greet { Person { name } } // 참고: dyn Trait — 런타임에 타입이 결정되는 트레잇 오브젝트 let greeters: Vec<Box<dyn Greet>> = vec![ Box::new(Person { name: "Alice".into() }), ];

impl Trait은 컴파일 타임에 구체 타입이 정해지고(정적 디스패치), dyn Trait은 실행 중에 결정됩니다(동적 디스패치). 지금은 "이런 게 있다" 정도만 기억해도 충분합니다.

실습: 인메모리 key-value 스토어

HashMap으로 데이터를 저장하고, Storage trait으로 "어떤 저장소든 갖춰야 할 인터페이스"를 정의해봅니다.

Storage trait + MemoryStore

use std::collections::HashMap; trait Storage { fn set(&mut self, key: String, value: String); fn get(&self, key: &str) -> Option<&String>; } struct MemoryStore { data: HashMap<String, String>, } impl Storage for MemoryStore { fn set(&mut self, key: String, value: String) { self.data.insert(key, value); } fn get(&self, key: &str) -> Option<&String> { self.data.get(key) } } // 나중에 파일 저장 버전(FileStore)을 추가해도 // Storage trait만 구현하면 동일하게 쓸 수 있어요.

5주차 · 컬렉션과 트레잇

흔한 실수

01
trait bound가 안 맞아서 나는 컴파일 에러 largest::<T>(&list)에서 T가 PartialOrd를 구현하지 않으면 "binary operation `>` cannot be applied" 에러가 납니다. 제네릭 함수 안에서 쓰는 연산자만큼 트레잇 바운드를 붙여야 해요.
02
iterator adaptor가 lazy라서 아무 일도 안 일어남 nums.iter().map(|n| println!("{n}")); 처럼 collect나 for 없이 끝내면 map 클로저가 아예 실행되지 않습니다. "unused `Map` that must be used" 경고가 힌트예요.
03
순회하면서 동시에 Vec을 수정 for x in &v { v.push(x); } 는 "cannot borrow `v` as mutable because it is also borrowed as immutable" 에러가 납니다. 순회가 끝난 뒤에 수정하거나 인덱스로 접근하세요.
04
HashMap에 insert한 뒤 원본 변수를 다시 사용 scores.insert(key, value) 는 key/value의 소유권을 가져갑니다. String을 넣은 뒤 원래 변수를 또 쓰려 하면 "value borrowed here after move" 에러가 발생해요. 필요하면 clone()하세요.
