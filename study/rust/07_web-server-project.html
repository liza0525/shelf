TOKIO

#[tokio::main]으로 비동기 실행하기

6주차에서 async/await 문법만 살짝 맛봤죠. 이번엔 tokio 런타임 위에서 진짜로 씁니다.

MAIN.RS

// #[tokio::main]이 런타임을 초기화하고 async fn main을 실행 가능하게 함 #[tokio::main] async fn main() { println!("서버 시작!"); let data = fetch_data().await; println!("받은 데이터: {}", data); } async fn fetch_data() -> String { // async fn은 호출 즉시 실행되지 않고 Future를 반환 // .await를 만나야 실제로 실행/대기함 (I/O 대기 등) tokio::time::sleep(std::time::Duration::from_millis(100)).await; "hello".to_string() }

AXUM

Router와 핸들러 함수

핸들러는 그냥 async fn입니다. 반환 타입이 IntoResponse를 구현하면 axum이 알아서 HTTP 응답으로 바꿔줍니다.

MAIN.RS

use axum::{routing::get, Router}; #[tokio::main] async fn main() { let app = Router::new() .route("/", get(root)) .route("/health", get(health_check)); let listener = tokio::net::TcpListener::bind("0.0.0.0:3000") .await .unwrap(); axum::serve(listener, app).await.unwrap(); } async fn root() -> &'static str { "Hello, Rust!" } async fn health_check() -> &'static str { "OK" }

ROUTING & EXTRACTOR

get/post/put/delete와 Path 추출

URL 경로의 :id는 Path<T>로, 쿼리스트링은 Query<T>로 타입 안전하게 꺼내 씁니다.

ROUTER.RS

use axum::{ extract::{Path, Query}, routing::get, Router, }; use std::collections::HashMap; let app = Router::new() .route("/boards", get(list_boards).post(create_board)) .route( "/boards/:id", get(get_board).put(update_board).delete(delete_board), ); async fn get_board(Path(id): Path<i64>) -> String { format!("게시글 #{id} 조회") } async fn list_boards(Query(params): Query<HashMap<String, String>>) -> String { format!("검색어: {:?}", params.get("q")) }

JSON EXTRACTOR

Json<T>로 요청 파싱, 응답 직렬화

#[derive(Deserialize)] / #[derive(Serialize)]로 struct와 JSON을 자동 변환합니다. FastAPI의 Pydantic 모델과 역할은 같지만, 타입이 컴파일 타임에 고정된다는 점이 다릅니다.

HANDLER.RS

use axum::{extract::Json, response::Json as JsonResponse}; use serde::{Deserialize, Serialize}; #[derive(Deserialize)] struct CreateBoard { title: String, content: String, } #[derive(Serialize)] struct Board { id: i64, title: String, content: String, } async fn create_board( Json(payload): Json<CreateBoard>, ) -> JsonResponse<Board> { let board = Board { id: 1, title: payload.title, content: payload.content, }; JsonResponse(board) }

SQLX

타입 안전한 DB 연동

sqlx의 핵심은 컴파일타임 쿼리 체크입니다. 빌드 시점에 실제 DB 스키마와 SQL을 대조해서, 컬럼명 오타나 타입 불일치를 빌드 실패로 잡아냅니다.

CONNECTION POOL

use sqlx::postgres::PgPoolOptions; #[tokio::main] async fn main() { let pool = PgPoolOptions::new() .max_connections(5) .connect("postgres://user:pass@localhost/board_db") .await .unwrap(); }

컴파일타임 쿼리 체크

let board = sqlx::query_as!( Board, "SELECT id, title, content FROM boards WHERE id = $1", id ) .fetch_one(&pool) .await?;

API DESIGN

board CRUD 엔드포인트 설계

Python/Node로 이미 만들어본 그 API입니다. 이번엔 axum + sqlx로 다시 만듭니다.

Create

Json<CreateBoard>로 요청 본문을 추출 → INSERT → 생성된 Board를 반환

Read

PUT /boards/:id

Path와 Json을 함께 추출 → UPDATE → 변경된 Board를 반환

DELETE /boards/:id

Delete

Path<i64>로 id 추출 → DELETE → 204 No Content 반환

COMPARE

FastAPI/Flask 대비 무엇이 다른가

가장 큰 차이는 타입 안전성과 컴파일타임 검증입니다.

FastAPI / Flask / Express

  • 타입 오류는 런타임에야 발견
  • ORM 쿼리 오타는 실행해봐야 앎
  • None/undefined 체크 깜빡하면 500 에러
  • 동시성은 프레임워크가 알아서 처리

axum + sqlx

  • 컴파일 타임에 타입 체크
  • SQL 오타·스키마 불일치는 빌드 실패로 잡힘
  • Option<T>/Result<T, E>로 누락 처리 강제
  • 소유권 덕분에 데이터 레이스 없는 동시성

ERROR HANDLING

IntoResponse로 에러를 HTTP 응답으로

핸들러가 Result<T, E>를 반환하고 E가 IntoResponse를 구현하면, axum이 자동으로 에러를 응답으로 바꿔줍니다. ?연산자로 전파하는 건 4주차와 똑같습니다.

ERROR.RS

use axum::{ http::StatusCode, response::{IntoResponse, Response}, Json, }; enum AppError { NotFound, Db(sqlx::Error), } impl IntoResponse for AppError { fn into_response(self) -> Response { let (status, message) = match self { AppError::NotFound => { (StatusCode::NOT_FOUND, "게시글을 찾을 수 없습니다".to_string()) } AppError::Db(e) => (StatusCode::INTERNAL_SERVER_ERROR, e.to_string()), }; (status, message).into_response() } } async fn get_board(Path(id): Path<i64>) -> Result<Json<Board>, AppError> { let board = sqlx::query_as!(Board, "SELECT * FROM boards WHERE id = $1", id) .fetch_optional(&pool) .await .map_err(AppError::Db)? .ok_or(AppError::NotFound)?; Ok(Json(board)) }
