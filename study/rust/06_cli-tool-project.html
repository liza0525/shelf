CLAP CRATE

커맨드라인 인자, 구조체로 선언하기

Python의 argparse가 파서 객체를 조립한다면, clap은 구조체에 derive 매크로를 붙이는 것만으로 파싱 규칙이 완성됩니다.

Cargo.toml

// clap의 derive 기능을 활성화 [dependencies] clap = { version = "4", features = ["derive"] }

src/main.rs

use clap::Parser; /// 인사 도구 — derive(Parser)가 파싱 로직을 자동 생성 #[derive(Parser, Debug)] #[command(name = "greet", version, about = "인사 도구")] struct Cli { /// 인사할 이름 (필수 인자) name: String, /// 인사 횟수 (-c, --count, 기본값 1) #[arg(short, long, default_value_t = 1)] count: u8, } fn main() { let cli = Cli::parse(); for _ in 0..cli.count { println!("안녕하세요, {}!", cli.name); } }

SUBCOMMANDS

mytool add / list / remove

실전 CLI 툴은 보통 여러 동작을 서브커맨드로 나눕니다. clap에서는 enum 하나로 이 구조를 표현합니다.

use clap::{Parser, Subcommand}; #[derive(Parser)] #[command(name = "mytool", version, about = "간단한 CLI 도구")] struct Cli { #[command(subcommand)] command: Commands, } #[derive(Subcommand)] enum Commands { /// 항목 추가 Add { text: String }, /// 전체 목록 출력 List, /// 항목 삭제 Remove { id: u32 }, } fn main() { let cli = Cli::parse(); match cli.command { Commands::Add { text } => println!("추가: {text}"), Commands::List => println!("목록 출력"), Commands::Remove { id } => println!("삭제: {id}"), } }

FILE I/O & ERROR

? 연산자로 깔끔하게 실패 전파하기

std::fs로 파일을 읽고 쓰되, 실패 가능성은 Result로 표현하고 ?로 위로 던집니다.

use std::fs; use anyhow::{Context, Result}; fn read_lines(path: &str) -> Result<Vec<String>> { // 파일 읽기 실패 시 에러에 경로 컨텍스트를 덧붙여 반환 let content = fs::read_to_string(path) .with_context(|| format!("파일을 읽을 수 없음: {path}"))?; Ok(content.lines().map(String::from).collect()) }

anyhow

빠르게 에러를 합치고 싶을 때

애플리케이션(바이너리) 코드에서 다양한 에러를 하나의 타입으로 다루고, .context()로 설명을 덧붙이기 좋음

thiserror

내 에러 타입을 정의하고 싶을 때

라이브러리 코드처럼 호출자가 에러 종류를 구분해야 할 때, derive(Error)로 커스텀 enum 정의

PROJECT 1

로그 파서: 세고 거르기

텍스트 로그 파일을 한 줄씩 읽어 특정 패턴이 몇 번 나오는지 세거나, 패턴이 포함된 줄만 골라냅니다.

use std::fs; use anyhow::Result; fn count_pattern(path: &str, pattern: &str) -> Result<usize> { let content = fs::read_to_string(path)?; let count = content .lines() .filter(|line| line.contains(pattern)) .count(); Ok(count) }

사용 예

$ logparser count access.log --pattern ERROR $ logparser filter access.log --pattern "500"

PROJECT 2

TODO 툴: 구조체 + JSON 영속화

add/list/done/remove 서브커맨드로 항목을 관리하고, serde로 구조체를 JSON으로 직렬화해 파일에 저장합니다.

use serde::{Deserialize, Serialize}; use std::fs; use anyhow::Result; #[derive(Serialize, Deserialize, Debug)] struct Todo { id: u32, text: String, done: bool, } fn save(todos: &[Todo], path: &str) -> Result<()> { let json = serde_json::to_string_pretty(todos)?; fs::write(path, json)?; Ok(()) } fn load(path: &str) -> Result<Vec<Todo>> { let content = fs::read_to_string(path)?; Ok(serde_json::from_str(&content)?) }

Cargo.toml — serde = { version = "1", features = ["derive"] }, serde_json = "1"

COMPARE

argparse 대비 clap

둘 다 강력하지만, 파싱 규칙을 "언제" 검증하느냐가 다릅니다.

Python · argparse

  • 런타임에 인자 규칙을 체크
  • 서브커맨드마다 add_parser() 반복 호출
  • 인자값은 기본적으로 문자열, 직접 변환 필요
  • 도움말 메시지를 수동으로 다듬어야 함

Rust · clap derive

  • 구조체 필드 타입 선언 = 파싱 + 검증
  • #[derive(Subcommand)] enum 하나로 정의
  • 컴파일 타임에 인자 타입 오류를 잡아냄
  • --help, --version 자동 생성
