STRUCT

struct — 데이터를 묶는 상자

여러 필드를 하나의 타입으로 묶는다는 점에서 Python의 클래스, Java의 필드 선언과 비슷해요. 다만 Rust의 struct는 기본적으로 데이터만 담고, 동작(메서드)은 별도의 impl 블록에서 정의해요.

CODE

struct User { username: String, email: String, active: bool, } // 필드 이름: 값 형태로 인스턴스를 만든다 let user1 = User { username: String::from("shelfy"), email: String::from("shelfy@rust.dev"), active: true, }; println!("{}", user1.username);

IMPL BLOCK

impl — 메서드와 연관 함수

&self를 받으면 인스턴스에 붙는 메서드, 받지 않으면 연관 함수(associated function)예요. Self::new(...)는 Java의 생성자·static factory method 자리를 대신하는 Rust식 관용구예요.

CODE

impl User { // 연관 함수 — self가 없다, 생성자처럼 사용 fn new(username: &str, email: &str) -> Self { Self { username: username.to_string(), email: email.to_string(), active: true, } } // 메서드 — &self로 인스턴스를 빌려온다 fn summary(&self) -> String { format!("{} <{}>", self.username, self.email) } } let user2 = User::new("lizzie", "lizzie@rust.dev"); println!("{}", user2.summary());

ENUM

enum — Java보다 훨씬 강력한 열거형

Java의 enum은 상수 목록에 가깝지만, Rust의 enum은 variant마다 서로 다른 데이터를 가질 수 있어요. 하나의 타입 안에 "값이 없음 / 값 하나 / 여러 필드"를 모두 표현할 수 있죠.

CODE

enum WebEvent { PageLoad, // 데이터 없는 variant KeyPress(char), // 값 하나를 담는 variant Click { x: i64, y: i64 }, // 구조체처럼 여러 필드를 담는 variant } fn handle(event: WebEvent) { match event { WebEvent::PageLoad => println!("페이지 로드됨"), WebEvent::KeyPress(c) => println!("키 입력: {}", c), WebEvent::Click { x, y } => println!("클릭 위치: ({}, {})", x, y), } }

OPTION<T>

Option<T> — null이 없는 세계

Rust에는 null이 없어요. "값이 없을 수 있음"은 Option<T> 타입으로 표현하고, 컴파일러가 이 가능성을 반드시 처리하게 강제해요.

PYTHON / JAVA

  • None / null이 어디서나 슬쩍 등장
  • user.name 접근 시 런타임에야 터짐
  • AttributeError / NullPointerException
  • Tony Hoare가 "십억 달러의 실수"라 부른 그것

RUST Option<T>

  • enum Option<T> { Some(T), None }
  • 값이 없을 수 있다는 게 타입 시그니처에 드러남
  • match/if let으로 처리 안 하면 컴파일 에러
  • NullPointerException 자체가 발생할 수 없음

RESULT<T, E>

Result<T, E> — 예외 대신 값

Rust에는 예외(exception)가 없어요. 실패할 수 있는 함수는 Result<T, E>를 리턴해서 "성공 아니면 실패"를 타입으로 드러내요.

PYTHON / JAVA

  • try/except: 뭐가 터질지 시그니처만 봐선 모름
  • Java checked exception도 놓치기 쉬움
  • 처리 잊으면 프로그램이 그냥 죽음

RUST Result<T, E>

  • enum Result<T, E> { Ok(T), Err(E) }
  • 실패 가능성이 함수 시그니처에 명시됨
  • 무시하면 컴파일러가 경고를 띄움

CODE

use std::num::ParseIntError; fn parse_age(input: &str) -> Result<u32, ParseIntError> { let age: u32 = input.parse()?; Ok(age) }

MATCH

match — exhaustive한 패턴 매칭

Python의 match-case보다 강력해요. variant, 범위, 구조 분해, 가드(guard) 조건까지 다루고, 모든 경우를 빠짐없이 처리해야 컴파일이 통과돼요.

CODE

enum Direction { North, South, East, West } fn describe(d: Direction) -> &'static str { match d { Direction::North => "북쪽", Direction::South => "남쪽", Direction::East => "동쪽", Direction::West => "서쪽", // variant 하나라도 빠지면 컴파일 에러! } } let n = 7; match n { 0 => println!("영"), 1..=5 => println!("1~5 사이"), n if n % 2 == 0 => println!("짝수: {}", n), _ => println!("그 외: {}", n), }

IF LET

if let — match의 축약형

"한 가지 패턴에만 관심 있고 나머지는 무시하고 싶을 때" match 대신 if let을 쓰면 코드가 훨씬 간결해져요.

CODE

let config_max: Option<u8> = Some(3); // match로 쓰면 다소 장황함 match config_max { Some(max) => println!("최댓값: {}", max), None => (), } // if let으로 축약 — "Some일 때만 관심 있음" if let Some(max) = config_max { println!("최댓값: {}", max); } else { println!("설정 없음"); }

? OPERATOR

? 연산자로 에러 전파하기

?는 Result가 Err면 그 자리에서 즉시 함수를 빠져나가 호출자에게 Err를 돌려주고, Ok면 안의 값을 꺼내줘요. try/except 블록 여러 겹을 대신하는 셈이에요.

CODE

use std::fs::File; use std::io::{self, Read}; fn read_username_from_file() -> Result<String, io::Error> { let mut file = File::open("username.txt")?; // 실패 시 즉시 Err 반환 let mut username = String::new(); file.read_to_string(&mut username)?; Ok(username) } match read_username_from_file() { Ok(name) => println!("사용자: {}", name), Err(e) => println!("읽기 실패: {}", e), }

실습 미리보기

파일 파싱 도구 — CSV → struct 벡터

지금까지 배운 struct, Result, ?를 한 번에 써먹는 실습이에요. CSV 한 줄을 파싱해서 실패하면 Err, 성공한 줄만 모아 Vec<Record>를 만들어요.

CODE

struct Record { name: String, age: u32, } fn parse_line(line: &str) -> Result<Record, String> { let parts: Vec<&str> = line.split(',').collect(); let name = parts.get(0).ok_or("이름 없음")?.to_string(); let age = parts.get(1) .ok_or("나이 없음")? .parse() .map_err(|_| "숫자가 아님")?; Ok(Record { name, age }) } fn parse_csv(text: &str) -> Vec<Record> { text.lines() .filter_map(|line| parse_line(line).ok()) .collect() }

흔한 실수

이런 실수는 하지 마세요

01
unwrap() 남발 Option이 None이거나 Result가 Err일 때 unwrap()은 그 자리에서 panic!으로 프로그램을 죽여요. 실무 코드에서는 match, if let, ?로 명시적으로 처리하세요.
02
match에서 _ 와일드카드 남용 모든 가지를 _로 뭉개면 exhaustiveness 체크의 장점이 사라져요. enum에 variant가 새로 추가돼도 컴파일러가 알려주지 않아요.
03
Result를 그냥 무시하기 #[must_use]로 표시된 Result를 받고도 아무 처리 안 하면 컴파일러가 경고해요. 경고를 무시하지 말고 반드시 에러 케이스를 다루세요.
04
데이터를 가진 variant를 값처럼 쓰기 WebEvent::KeyPress(c)처럼 데이터를 담은 variant는 match/if let으로 구조 분해해야 안의 값을 꺼낼 수 있어요. 그냥 비교하면 컴파일 에러가 나요.

SUMMARY

4주차 요약

STRUCT / IMPL

데이터 + 동작 분리

struct는 필드만, impl은 메서드와 Self::new 연관 함수를 담당해요.

ENUM

variant마다 다른 데이터

Java enum보다 강력하게, 하나의 타입 안에서 여러 형태를 표현해요.

OPTION / RESULT

null과 예외를 대신하는 타입

있음/없음, 성공/실패를 타입으로 드러내고 컴파일러가 처리를 강제해요.

MATCH

exhaustive 패턴 매칭

모든 경우를 빠짐없이 다뤄야 컴파일이 통과돼요.

IF LET

match의 간결한 축약형

한 가지 패턴에만 관심 있을 때 코드를 짧게 써요.

? 연산자

에러를 위로 전파

Err면 즉시 반환, Ok면 값을 꺼내는 체이닝을 가능하게 해요.
