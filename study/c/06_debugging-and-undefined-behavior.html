디버깅과 Undefined Behavior

← → 키로 페이지를 넘길 수 있어요.

이번 주 목표

"컴파일은 됐는데
왜 죽는지 모르겠다"를 없앤다

Python에선 예외가 스택 트레이스를 친절하게 알려줬지만, C는 그냥 조용히 죽거나 — 더 무섭게도 — 조용히 틀린 값을 낸다. 이번 주는 그 침묵을 깨는 도구들을 다룬다.

Debugger

gdb

실행 중인 프로그램을 멈춰서 상태를 들여다본다.

Language Pitfall

Undefined Behavior

C 표준이 "결과를 보장하지 않는다"고 선언한 영역.

Compiler

경고 + Sanitizer

컴파일 타임/런타임에 문제를 미리 잡아낸다.

Static Analysis

clang-tidy / cppcheck

실행 없이 소스 코드만 보고 의심 패턴을 찾는다.

Why gdb

print 디버깅은 확장이 안 된다

Python에서 pdb.set_trace() 한 줄로 되던 일이, C에서는 printf를 심고 재컴파일하고 재현하는 루프가 되기 쉽다. gdb는 그 루프를 없앤다.

printf 디버깅

  • 추측한 지점에만 로그를 심을 수 있다
  • 한 줄 추가할 때마다 재컴파일해야 한다
  • 세그폴트 나면 그 순간 정보가 사라진다
  • 변수 하나 바뀌는 순간을 잡기 어렵다

gdb 디버깅

  • 실행 중 아무 지점에나 멈출 수 있다
  • 멈춘 채로 임의의 표현식을 평가한다
  • 크래시 순간의 스택을 그대로 보존한다
  • watch로 값이 바뀌는 순간 자동 정지

gdb 기초

알아야 할 명령어 다섯 개

gdb는 명령어가 수백 개지만, 실전에서는 이 다섯 개로 8할이 해결된다.

break (b)

중단점 설정

break main
break buggy.c:42
지정한 함수/줄에서 실행을 멈춘다.

step / next (s / n)

한 줄씩 실행

step은 함수 호출이 있으면 내부로 들어가고, next는 호출을 건너뛰고 다음 줄로 간다.

print (p)

값 출력

print x
print arr[3]
print *ptr
변수든 표현식이든 그 자리에서 평가한다.

backtrace (bt)

호출 스택 출력

지금 이 지점까지 어떤 함수들이 어떤 순서로 호출됐는지 전부 보여준다.

watch

변수 감시

지정한 변수의 값이 바뀌는 그 순간 자동으로 멈춘다. "언제 이 값이 깨지는가"를 찾을 때 필수.

실전 세션

gdb 세션 한 번 따라가보기

디버그 심볼 없이는 gdb도 무력하다. 항상 -g와 함께, 최적화는 -O0으로 컴파일한다.

# 1. 디버그 심볼 포함 + 최적화 끄고 컴파일 $ gcc -g -O0 -o buggy buggy.c # 2. gdb로 실행 파일 열기 $ gdb ./buggy (gdb) break main Breakpoint 1 at 0x1169: file buggy.c, line 8. (gdb) run Starting program: /home/user/buggy Breakpoint 1, main () at buggy.c:8 8 int total = sum(arr, 5); (gdb) step sum (arr=0x7fffffffe3a0, n=5) at buggy.c:15 15 int result = 0; (gdb) watch result Hardware watchpoint 2: result (gdb) continue Hardware watchpoint 2: result Old value = 0 New value = 1 sum (arr=0x7fffffffe3a0, n=5) at buggy.c:17 (gdb) print n $1 = 5 (gdb) backtrace #0 sum (arr=0x7fffffffe3a0, n=5) at buggy.c:17 #1 0x0000555555555189 in main () at buggy.c:8

Undefined Behavior

"동작할 수도 있다"가 가장 위험한 말

UB는 에러도, 경고도 아니다. 표준이 "이 경우 아무 결과나 나와도 된다"고 선언한 영역이라 지금 잘 돌아간다고 안전하다는 뜻이 아니다.

Overflow

정수 오버플로우

int x = INT_MAX;
x = x + 1; // UB
signed 오버플로우는 UB. unsigned는 wraparound가 정의돼 있어 안전하다.

Uninitialized

초기화되지 않은 변수

int x;
if (x > 0) { ... } // UB
스택에 남아있던 쓰레기 값을 읽는다. 매번 다른 값일 수 있다.

Out-of-bounds

배열 경계 밖 접근

int arr[5];
arr[5] = 10; // UB
C는 경계 검사를 안 한다. 옆 메모리를 조용히 덮어쓸 수 있다.

Type Punning

strict aliasing 위반

float f = 3.14f;
int i = *(int*)&f; // UB
다른 타입 포인터로 재해석해 읽으면 UB. memcpy로 바이트를 복사해야 한다.

The Real Danger

컴파일러는 UB가 일어나지 않는다고 가정한다

UB는 "런타임에 이상한 값이 나온다"로 끝나지 않는다. 최적화 컴파일러는 UB가 절대 일어나지 않는다고 전제하고, 그 전제를 이용해 코드를 통째로 지워버릴 수 있다.

// 오버플로우를 감지하려는 의도로 짠 코드 if (x + 1 < x) { // "오버플로우 났다" 처리... 하려 했지만 handle_overflow(); } // -O2 컴파일러의 추론: // "signed overflow는 UB → 절대 안 일어난다고 가정해도 됨 // → x + 1 < x 는 항상 거짓 → 이 if문은 죽은 코드다" // 결과: handle_overflow() 호출이 통째로 삭제될 수 있다

-O0에서는 멀쩡히 동작하다가 릴리즈 빌드(-O2)에서만 재현되는 버그의 전형적인 원인이다.

방어선 1 — 컴파일러

경고와 Sanitizer를 최대로 켠다

기본 옵션만으로 컴파일하는 건 안전벨트를 풀고 운전하는 것과 같다.

$ gcc -Wall -Wextra -Wpedantic \ -fsanitize=address,undefined \ -g -O0 \ -o debug_build main.c ## -Wall : 흔한 실수 관련 경고 (미초기화, 비교 실수 등) ## -Wextra : -Wall보다 더 꼼꼼한 추가 경고 ## -Wpedantic : ISO C 표준 위반까지 엄격하게 검사 ## -fsanitize=address : out-of-bounds, use-after-free 등 메모리 오류 탐지 (ASan) ## -fsanitize=undefined : 정수 오버플로우, 정렬 오류 등 UB 런타임 탐지 (UBSan)

방어선 2 — 정적 분석

실행하지 않고도 버그를 찾는다

Sanitizer는 "실행된 경로"만 검사한다. 정적 분석 도구는 실행 없이 소스 전체를 훑어 의심스러운 패턴을 찾아준다 — 상호 보완적이다.

LLVM 기반

clang-tidy

$ clang-tidy main.c -- -Wall
버그 패턴은 물론 "더 현대적인 C 스타일" 제안까지 해준다. 검사 항목을 세밀하게 켜고 끌 수 있다.

가볍고 빠름

cppcheck

$ cppcheck --enable=all main.c
오탐(false positive)이 적어 CI 파이프라인에 부담 없이 넣기 좋다.

실습 미리보기

일부러 심은 버그를 사냥한다

오늘 실습은 UB가 심어진 코드를 받아, 이 시간에 배운 도구들만으로 원인을 특정하고 고치는 것이다.

Step 1

버그 심기

오버플로우·미초기화·out-of-bounds가 섞인 코드를 받는다

Step 2

gdb로 재현

breakpoint와 step으로 크래시 직전 상태를 관찰한다

Step 3

sanitizer로 특정

ASan/UBSan 리포트로 정확한 줄과 원인을 확인한다

Step 4

수정

원인 제거 후 재컴파일, sanitizer 통과까지 확인한다

흔한 실수

디버깅 환경에서 자주 걸려 넘어지는 지점

01
-g 없이 gdb 켜기 심볼 없는 릴리즈 바이너리를 gdb로 열면 함수명 대신 "??"만 보인다. 항상 -g -O0으로 디버그 빌드를 따로 만든다.
02
sanitizer는 느리니까 마지막에만 개발 내내 꺼두고 배포 직전에만 돌리면, 몇 주 전에 심긴 버그의 원인을 찾기가 훨씬 어려워진다. 평소 개발 빌드에 기본으로 켜둔다.
03
경고를 "그냥 warning"으로 무시 -Wall이 가리키는 미초기화 변수, 타입 불일치 경고를 넘기다가 그대로 런타임 버그가 된다.
04
print문 재컴파일 루프 printf 추가 → 재컴파일 → 재현을 반복하는 대신, gdb의 watch/backtrace 한 번이면 끝나는 경우가 많다.

Wrap Up

이번 주 챙길 것

①

gdb 다섯 개 명령어

break, step/next, print, backtrace, watch — 이것만으로 대부분의 크래시를 추적할 수 있다

②

UB는 "가끔 틀린 값"이 아니라 "무엇이든 가능"

오버플로우, 미초기화, out-of-bounds, 타입 punning — 컴파일러는 이런 코드를 예상대로 최적화하지 않는다

③

-Wall -Wextra -Wpedantic -fsanitize=address,undefined는 기본값

여기에 clang-tidy / cppcheck로 실행 없이도 한 번 더 걸러낸다

다음 주(7~8주차)는 이 모든 걸 동원해 미니 셸을 직접 구현한다.
