STRUCT LAYOUT

struct의 메모리 배치와 패딩

Python의 클래스 인스턴스는 필드가 딕셔너리처럼 흩어져 있지만, C의 struct는 필드가 메모리에 연속으로 배치돼요. 단, 컴파일러가 각 필드를 정렬(alignment) 요구사항에 맞추려고 중간에 패딩 바이트를 끼워 넣어요.

CODE

struct Sample { char a; // 1 byte, offset 0 int b; // 4 byte, int는 4바이트 정렬 필요 → offset 1~3은 패딩 char c; // 1 byte, offset 8 }; // offset 9~11도 패딩 (구조체 전체 크기도 4의 배수여야 함) printf("%zu\n", sizeof(struct Sample)); // 12 (1+3패딩 + 4 + 1+3패딩) // 필드 순서만 바꿔도 패딩이 줄어든다 struct Packed { int b; // offset 0 char a; // offset 4 char c; // offset 5, 뒤에 패딩 2바이트만 }; printf("%zu\n", sizeof(struct Packed)); // 8

UNION

union — 필드들이 메모리를 공유한다

struct는 모든 필드가 각자 자기 공간을 갖지만, union은 가장 큰 필드 크기만큼만 메모리를 할당하고 모든 필드가 그 공간을 겹쳐 쓴다. Python에는 직접 대응하는 개념이 없어요 — 굳이 비유하면 "타입에 따라 다르게 해석되는 같은 바이트 뭉치".

STRUCT — 각자 공간

  • int i, float f, char str[20]을 모두 동시에 보유
  • sizeof는 각 필드 크기의 합 + 패딩 (보통 28)
  • 모든 필드에 언제든 동시 접근 가능
  • 메모리를 더 쓰는 대신 안전함

UNION — 공간 공유

  • int i, float f, char str[20]이 같은 주소를 공유
  • sizeof는 가장 큰 필드 크기 (str[20] 기준 20)
  • 마지막에 쓴 필드만 유효한 값
  • 메모리는 아끼지만 어떤 필드가 유효한지 직접 추적해야 함

POINTER · ->

구조체 포인터와 -> 연산자

p->field는 (*p).field의 축약형이에요. 값으로 포함된 중첩 구조체는 .으로, 포인터로 가리키는 구조체는 ->로 접근한다는 규칙만 기억하면 돼요.

CODE

struct Point { int x, y; }; struct Rect { struct Point origin; // 중첩 구조체 — 값으로 포함 (포인터 아님) int width, height; }; void move_point(struct Point *p, int dx, int dy) { p->x += dx; // (*p).x += dx; 와 완전히 동일 p->y += dy; } struct Rect r = { .origin = {0, 0}, .width = 10, .height = 20 }; move_point(&r.origin, 5, 5); // 중첩 필드의 주소를 넘긴다 printf("(%d, %d)\n", r.origin.x, r.origin.y); // (5, 5) — . 으로 접근

SINGLY LINKED LIST

연결 리스트 — 자기 자신을 가리키는 struct

Python의 list는 언어가 배열을 관리해 주지만, C에서는 malloc으로 노드를 직접 만들고 포인터로 이어붙여야 해요. struct 안에 자기 타입을 가리키는 포인터를 두면 재귀적 구조가 돼요.

CODE

typedef struct Node { int value; struct Node *next; // typedef 이름 Node는 아직 완성 전이라 struct Node로 참조 } Node; Node *push_front(Node *head, int value) { Node *node = malloc(sizeof(Node)); if (!node) return head; // malloc 실패 시 원래 리스트 그대로 반환 node->value = value; node->next = head; // 새 노드가 기존 head를 가리킨다 return node; // 새로운 head를 호출자에게 돌려준다 }

DOUBLY LINKED LIST

이중 연결 리스트로 확장

prev 포인터 하나만 추가하면 양방향 순회가 가능해져요. head/tail을 함수 안에서 바꾸려면 포인터의 포인터(이중 포인터)가 필요해요 — 값 전달만으론 호출자의 변수가 안 바뀌기 때문.

CODE

typedef struct DNode { int value; struct DNode *prev; struct DNode *next; } DNode; DNode *dnode_create(int value) { DNode *node = malloc(sizeof(DNode)); node->value = value; node->prev = NULL; node->next = NULL; return node; } void dlist_push_back(DNode **head, DNode **tail, int value) { DNode *node = dnode_create(value); if (*tail == NULL) { // 빈 리스트였던 경우 *head = *tail = node; } else { node->prev = *tail; (*tail)->next = node; *tail = node; } }

STACK ON LINKED LIST

연결 리스트로 구현하는 스택

배열 기반 스택은 크기를 미리 정하거나 realloc이 필요하지만, 연결 리스트 기반 스택은 top 포인터 하나만 관리하면 돼요. push/pop 모두 O(1)이에요.

CODE

typedef struct { DNode *top; } Stack; void stack_push(Stack *s, int value) { DNode *node = dnode_create(value); node->prev = s->top; if (s->top) s->top->next = node; s->top = node; } int stack_pop(Stack *s, int *out) { if (!s->top) return 0; // 빈 스택 — 실패 DNode *old_top = s->top; *out = old_top->value; s->top = old_top->prev; if (s->top) s->top->next = NULL; free(old_top); // 꺼낸 노드는 반드시 해제 return 1; }

TRAVERSAL

순회와 전체 해제 함수

연결 리스트를 다 쓰고 나면 노드를 하나씩 free해야 해요. 이때 free하기 전에 다음 포인터를 미리 저장해두지 않으면, free된 메모리를 다시 읽는 use-after-free가 발생해요.

CODE

void stack_print(const Stack *s) { for (DNode *cur = s->top; cur != NULL; cur = cur->prev) { printf("%d ", cur->value); } printf("\n"); } void stack_free_all(Stack *s) { DNode *cur = s->top; while (cur != NULL) { DNode *next = cur->prev; // free 전에 다음 노드 주소를 미리 저장 free(cur); cur = next; } s->top = NULL; // dangling pointer 방지 }

흔한 실수

구조체 & 연결 리스트, 이렇게 틀린다

01
free 순서 실수로 리스트가 끊긴다 cur를 free한 뒤에 cur->next로 다음 노드를 찾으려 하면 이미 해제된 메모리를 읽는 use-after-free가 발생해요. next는 free 전에 반드시 미리 저장하세요.
02
패딩을 가정하고 memcpy/캐스팅을 사용한다 구조체를 통째로 파일에 쓰거나 다른 구조체로 캐스팅하면, 패딩 바이트 값은 정의되지 않고 컴파일러·플랫폼마다 패딩 크기가 달라질 수 있어요. 필드 단위로 직렬화하는 편이 안전해요.
03
head 포인터를 값으로 넘겨 함수 밖에서 안 바뀐다 Node *head를 그냥 값으로 받으면 함수 안에서 head를 바꿔도 호출자의 변수는 그대로예요. head를 함수 안에서 갱신하려면 Node **head 이중 포인터가 필요해요.
04
union에서 마지막에 쓴 필드가 아닌 다른 필드를 읽는다 union은 어떤 필드가 마지막으로 유효하게 쓰였는지 스스로 기억하지 않아요. int로 써놓고 float로 읽으면 값이 아니라 비트 패턴을 재해석하게 돼요 — 별도의 태그(enum) 필드로 추적하세요.

힌트: 포인터도 값으로 전달되면 복사본이라, 원본 변수를 바꾸려면 그 주소가 필요함

4주차 요약

STRUCT / PADDING

필드마다 정렬 요구가 다르다

컴파일러가 정렬을 맞추려고 패딩을 끼워 넣어요. 필드 순서를 바꾸면 크기를 줄일 수 있어요.

UNION

메모리를 공유하는 필드들

가장 큰 필드 크기만큼만 할당되고, 마지막에 쓴 필드만 유효해요.

-> 연산자

(*p).field의 축약형

값으로 포함된 중첩 구조체는 ., 포인터가 가리키는 구조체는 ->로 접근해요.

연결 리스트

자기 참조 struct + malloc

단일은 next 하나, 이중은 prev/next 두 개로 양방향 순회를 지원해요.

스택 (LIFO)

top 포인터만 관리

연결 리스트 기반이면 크기 제한 없이 push/pop이 모두 O(1)이에요.

메모리 원칙

free 전에 다음 노드부터 저장

free 순서를 지키고, head 갱신에는 이중 포인터를 쓰는 습관을 들이세요.
