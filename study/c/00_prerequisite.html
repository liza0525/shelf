C 시작 전 환경 설정

VSCode 기준, Windows / Linux / macOS 차이를 항목별로 정리합니다. ← → 키로 페이지를 넘길 수 있어요.

들어가며

C는 OS마다 "당연한 것"이 다르다

Python은 인터프리터 하나만 있으면 어디서든 똑같이 돌아가지만, C는 컴파일러와 OS의 시스템 콜에 직접 의존해요. 특히 7~8주차 미니 셸 프로젝트는 `fork`/`pipe`/`dup2` 같은 POSIX API를 쓰기 때문에, OS 선택이 처음부터 중요합니다.

주의

  • Windows 네이티브에는 fork()가 없음
  • MSVC는 이 커리큘럼의 gcc 문법과 다르게 동작
  • Apple Silicon Mac은 valgrind 미지원

권장

  • Windows → WSL2(Ubuntu) 사용
  • Linux → 배포판 기본 gcc/gdb 그대로 사용
  • macOS → Xcode CLT + Docker(선택)로 valgrind 보완

준비물

설치할 4가지

01
VSCode 에디터 겸 터미널·디버거 통합 환경.
02
C/C++ 확장 팩 문법 하이라이트, 자동완성, 디버거 연동.
03
gcc/clang + make + gdb + valgrind 컴파일러, 빌드 도구, 디버거, 메모리 검사 도구.
04
터미널 (OS별로 다름) Windows는 WSL2, Linux/macOS는 기본 셸 그대로.

VSCode 설치

OS별 설치 방법

설치 파일은 code.visualstudio.com에서 받거나 패키지 매니저를 쓸 수 있어요.

Windows

  • 공식 installer(.exe) 실행
  • "PATH에 추가" 체크박스 반드시 체크
winget install Microsoft.VisualStudioCode

Linux

  • .deb/.rpm 또는 apt 저장소 등록
  • code 명령어 자동 등록됨
sudo apt install code # 또는 snap install code --classic

macOS

  • .dmg를 Applications로 드래그
  • code 명령은 팔레트에서 별도 등록 필요
brew install --cask visual-studio-code # 이후 Cmd+Shift+P # "Shell Command: Install 'code'"

VSCode 확장

C 개발에 필요한 확장

아래 확장은 OS 상관없이 동일하게 설치합니다. Windows에서 WSL을 쓰면 확장 하나가 더 필요해요.

ms-vscode.cpptools

문법 하이라이트, IntelliSense, 디버거 연동의 기본 확장.

컴파일러 · 도구 설치

gcc / make / gdb / valgrind

Windows는 WSL2 안에서 Linux와 동일한 명령을 그대로 씁니다.

Windows (WSL2)

  • PowerShell(관리자)에서 WSL 설치
  • 이후 Ubuntu 안에서 Linux 명령 그대로
wsl --install -d Ubuntu # 재부팅 후 Ubuntu 터미널에서 sudo apt update sudo apt install build-essential gdb valgrind

Linux

  • Debian/Ubuntu 계열
  • Fedora 계열은 dnf 사용
sudo apt install build-essential gdb valgrind # Fedora: sudo dnf install gcc gdb make valgrind

macOS

  • gcc는 clang의 별칭
  • gdb는 코드사이닝 문제로 lldb 권장
xcode-select --install brew install make # valgrind는 Apple Silicon 미지원 # → lldb 또는 Docker Linux 컨테이너 사용

설치 확인

버전 체크로 마무리

아래 명령이 오류 없이 버전을 출력하면 준비가 끝난 거예요.

터미널 (WSL / Linux / macOS 공통)

gcc --version make --version gdb --version # macOS는 gdb 대신 lldb --version

gcc --version 출력에 "Apple clang"이 보이면 정상 (macOS는 gcc가 clang의 별칭이기 때문).

환경 변수

PATH가 깨졌을 때 여기를 본다

Windows

  • WSL2 사용 시 Windows PATH 개념 자체가 무관
  • 네이티브 MSYS2 쓸 경우 시스템 속성 GUI에서 편집

Linux

  • ~/.bashrc 또는 ~/.profile에 export PATH
  • apt로 설치하면 대부분 자동으로 PATH에 잡힘

macOS

  • ~/.zshrc (기본 셸이 zsh)
  • Homebrew 경로: Apple Silicon은 /opt/homebrew/bin
  • Intel Mac은 /usr/local/bin (경로가 다름 주의)

VSCode 통합 터미널

기본 셸을 무엇으로 맞출까

VSCode 하단 터미널 패널에서 여는 기본 셸이 OS마다 다릅니다.

Windows

기본 터미널을 Ubuntu(WSL)로 지정

Ctrl+Shift+P → "Terminal: Select Default Profile" → Ubuntu (WSL) 선택. PowerShell로 열려 있으면 fork 관련 실습이 전부 실패합니다.

Linux / macOS

bash 또는 zsh 그대로 사용

별도 설정 없이 시스템 기본 셸이 그대로 열림. macOS 기본은 zsh, 대부분의 Linux 배포판 기본은 bash.

자주 걸리는 문제

OS별 트러블슈팅

Win
fork() 관련 함수가 컴파일이 안 됨 MSYS2/MinGW 네이티브 환경에서는 정상. WSL2를 안 쓰고 있다는 신호이니 터미널을 Ubuntu(WSL)로 바꾼다.
Linux
valgrind: command not found build-essential에는 valgrind가 포함되지 않음. 별도로 apt/dnf install valgrind 필요.
macOS
gdb 실행 시 권한 오류 macOS는 코드사이닝 없는 gdb 실행을 막음. lldb를 쓰거나 gdb에 자체 서명 인증서를 설정해야 함.
공통
-Wall -Wextra 경고가 OS마다 다르게 뜸 libc 구현(glibc/musl/BSD libc)이 달라 경고 종류가 조금씩 다를 수 있음. 정상적인 차이이니 당황하지 않는다.

디버거 연동

launch.json, 한 번만 맞추면 끝

C/C++ 확장이 gdb(Linux/WSL) 또는 lldb(macOS)를 자동으로 찾아 연결해요. 디버그 패널의 "Add Configuration"으로 생성합니다.

.vscode/launch.json (핵심 필드)

{ // Linux/WSL: "type": "cppdbg", "MIMode": "gdb" // macOS: "type": "cppdbg", "MIMode": "lldb" "program": "${workspaceFolder}/main", "stopAtEntry": false }

실습

VSCode에서 첫 C 프로그램 실행하기

01
빈 폴더 생성 후 code . 로 VSCode 열기 터미널에서 mkdir hello-c && cd hello-c && code .
02
main.c 작성 #include <stdio.h> 후 printf("Hello, C!\n")를 담은 main 함수.
03
통합 터미널에서 컴파일/실행 gcc -Wall -Wextra main.c -o main && ./main
04
F5로 디버거 실행해보기 중단점을 찍고 F5 → 변수 값이 디버그 패널에 보이면 연동 성공.

마무리

1주차 들어가기 전 셀프 체크

gcc --version, make --version이 오류 없이 출력되나요?

Windows라면 WSL(Ubuntu) 터미널에서 확인해야 해요.

VSCode 통합 터미널의 기본 셸이 올바르게 지정되어 있나요?

Windows는 Ubuntu(WSL), macOS/Linux는 기본 셸 그대로.

F5로 중단점 디버깅이 정상 동작하나요?

launch.json의 MIMode가 gdb/lldb 중 맞는 쪽인지 확인하세요.

내 OS에서 valgrind 대안이 무엇인지 알고 있나요?

Apple Silicon Mac은 valgrind 대신 lldb 또는 Docker Linux 컨테이너를 씁니다.
