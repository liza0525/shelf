Stack vs Heap

해제를 누가 책임지는가

Python의 모든 객체는 사실 힙에 있고, 참조 카운트/GC가 해제 시점을 알아서 숨겨줍니다. C는 스택은 자동, 힙은 수동 — 이 경계를 항상 의식해야 합니다.

스택 — 자동 해제

  • 함수 리턴 시 프레임과 함께 자동 소멸
  • 할당/해제가 매우 빠름 (포인터 이동)
  • 크기를 컴파일 타임에 알아야 함
  • 지역 변수, 함수 인자가 여기 위치

힙 — 수동 해제

  • malloc/free를 직접 짝지어야 함
  • 런타임에 크기를 결정할 수 있음
  • 함수를 벗어나도 살아있음
  • 안 지우면 누수, 두 번 지우면 크래시

malloc & calloc

할당은 실패할 수 있다

malloc/calloc은 실패하면 NULL을 반환합니다. Python의 리스트/딕셔너리는 메모리가 없으면 예외를 던지지만, C는 아무 신호 없이 NULL만 돌려줄 뿐입니다.

int *arr = malloc(/* 개수 */ 10 * sizeof(int)); // 반환값을 반드시 체크한다 if (arr == NULL) { fprintf(stderr, "malloc 실패\n"); exit(EXIT_FAILURE); } // calloc은 개수와 원소 크기를 따로 받고, 0으로 초기화까지 해준다 int *zeros = calloc(10, sizeof(int)); if (zeros == NULL) { fprintf(stderr, "calloc 실패\n"); exit(EXIT_FAILURE); } // zeros[0] ~ zeros[9]는 모두 0이 보장됨 (malloc은 보장 안 함!)

realloc & free

realloc은 새 주소를 줄 수 있다

realloc은 확장이 불가능하면 새 블록을 잡고 내용을 복사한 뒤 원본을 해제합니다. 그래서 반환값을 임시 변수로 받아야 안전합니다.

int *arr = malloc(4 * sizeof(int)); if (arr == NULL) exit(EXIT_FAILURE); // 잘못된 예: 실패하면 arr 자체가 NULL로 덮여 원본 주소를 잃는다 // arr = realloc(arr, 8 * sizeof(int)); // 절대 이렇게 쓰지 말 것 // 올바른 예: 임시 포인터로 받아서 성공 여부를 먼저 확인 int *tmp = realloc(arr, 8 * sizeof(int)); if (tmp == NULL) { fprintf(stderr, "realloc 실패, 기존 arr은 아직 유효함\n"); free(arr); exit(EXIT_FAILURE); } arr = tmp; free(arr); arr = NULL; // free 직후 관용적으로 NULL 대입

Skip the NULL Check

체크를 생략하면 벌어지는 일

Python에서 익숙한 "에러는 예외로 날아온다"는 가정이 C에는 없습니다. NULL 포인터를 그냥 역참조하면 세그폴트로 프로세스가 즉사합니다.

// 메모리가 부족하거나 요청 크기가 비정상적으로 큰 상황이라고 가정 int *huge = malloc((size_t)-1); // 사실상 항상 실패 huge[0] = 42; // NULL 역참조 → Segmentation fault (core dumped) // gdb/valgrind 없이 크래시 지점만 봐서는 원인 파악이 어렵다 // 반환값 체크 한 줄이면 막을 수 있었던 크래시

핵심

malloc 계열 함수는 항상 실패 가능성을 전제로 코드를 짜야 합니다.

Double Free & Use-After-Free

지워진 메모리를 다시 만지면

free는 메모리를 "반납 가능" 상태로 표시할 뿐, 포인터 값 자체는 그대로 남습니다. 이 포인터를 계속 쓰거나 두 번 해제하면 정의되지 않은 동작(UB)이 시작됩니다.

int *p = malloc(sizeof(int)); *p = 10; free(p); free(p); // 버그: double free — 힙 메타데이터 손상, 크래시 가능 int *q = malloc(sizeof(int)); free(q); printf("%d\n", *q); // 버그: use-after-free — 이미 반납된 메모리를 읽음 *q = 99; // 다른 malloc이 재사용 중인 메모리를 덮어쓸 수도 있음

Dangling Pointer

함수가 끝나면 가리킬 곳이 없다

Python에서 클로저가 변수를 캡처하듯 함수 밖으로 참조를 들고 나가는 건 흔한 일이지만, C에서 스택 지역 변수의 주소를 반환하면 그 메모리는 이미 회수된 상태입니다.

int *make_dangling(void) { int local = 42; return &local; // 버그: local은 스택 변수, 함수 종료와 동시에 소멸 } int main(void) { int *p = make_dangling(); printf("%d\n", *p); // dangling pointer 역참조 — 값이 우연히 맞을 수도, 크래시일 수도 // 올바른 방법: 힙에 할당해서 소유권을 호출자에게 넘긴다 // int *heap_val = malloc(sizeof(int)); *heap_val = 42; return heap_val; }

valgrind

눈에 안 보이는 버그를 잡아내기

컴파일은 되고 실행도 되지만 조용히 메모리를 새고 있는 코드 — valgrind는 이런 문제를 실행 중에 추적해서 리포트로 보여줍니다.

$ gcc -g -O0 -o prog prog.c $ valgrind --leak-check=full --track-origins=yes ./prog ==12345== HEAP SUMMARY: ==12345== in use at exit: 40 bytes in 1 blocks ==12345== total heap usage: 3 allocs, 2 frees, 1,064 bytes allocated ==12345== ==12345== 40 bytes in 1 blocks are definitely lost in loss record 1 of 1 ==12345== at 0x4C3...: malloc (vg_replace_malloc.c:299) ==12345== by 0x1091A9: main (prog.c:14) ==12345== ==12345== LEAK SUMMARY: ==12345== definitely lost: 40 bytes in 1 blocks

읽는 법

"definitely lost"는 free를 잊은 것, 스택 트레이스가 어느 malloc인지 정확히 짚어줍니다.

실습 미리보기

가변 크기 동적 배열 만들기

Python의 list.append()가 내부적으로 하는 일을 직접 구현합니다. 용량이 꽉 차면 realloc으로 두 배 늘리고, 다 쓰면 확실히 free합니다.

typedef struct { int *data; size_t size; // 현재 원소 개수 size_t capacity; // 할당된 용량 } IntVec; void vec_push(IntVec *v, int value) { if (v->size == v->capacity) { size_t new_cap = v->capacity == 0 ? 4 : v->capacity * 2; int *tmp = realloc(v->data, new_cap * sizeof(int)); if (tmp == NULL) exit(EXIT_FAILURE); v->data = tmp; v->capacity = new_cap; } v->data[v->size++] = value; } void vec_free(IntVec *v) { free(v->data); v->data = NULL; v->size = v->capacity = 0; } // $ valgrind --leak-check=full ./vec_demo // "All heap blocks were freed -- no leaks are possible" 가 목표

Common Pitfalls

자주 걸려 넘어지는 지점들

01
free 후 포인터를 그대로 쓴다 free(p)는 p의 값을 바꾸지 않는다. free 직후 p = NULL로 초기화하는 습관이 필요하다.
02
realloc 반환값을 원본 포인터에 바로 덮어쓴다 실패 시 NULL이 원본을 덮어써서 기존 메모리 주소를 영영 잃고 누수가 발생한다.
03
sizeof 계산을 잘못한다 malloc(sizeof(p)) vs malloc(sizeof(*p)) — 포인터 자체 크기와 가리키는 타입 크기를 혼동한다.
04
같은 블록을 두 번 free한다 구조체 안에 같은 포인터가 두 곳에서 참조될 때, 소유권을 누가 갖는지 명확히 하지 않으면 발생한다.
05
에러 경로에서 free를 빼먹는다 malloc 여러 번 이후 중간에 실패해서 조기 반환할 때, 이미 할당된 블록들의 해제를 잊는다.

Summary

이번 주 기억할 것

스택은 자동, 힙은 수동
malloc/calloc/realloc은 항상 NULL 체크
realloc은 임시 변수로 받고, free 후엔 포인터를 NULL로
valgrind는 컴파일러가 못 잡는 런타임 메모리 버그의 안전망
