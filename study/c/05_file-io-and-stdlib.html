fopen / fread / fwrite / fclose

Python의 open()은 실패하면 예외를 던지지만, fopen()은 NULL을 반환할 뿐입니다. try/except가 없으니 직접 체크해야 해요.

기본 패턴

FILE *fp = fopen("data.txt", "r"); // with open(...) as f: 같은 자동 정리는 없다 — NULL 체크는 필수 if (fp == NULL) { perror("fopen 실패"); return 1; } char buf[256]; size_t n = fread(buf, 1, sizeof buf, fp); // fread는 실패와 EOF를 반환값만으로 구분하지 못한다 if (n < sizeof buf && ferror(fp)) { fprintf(stderr, "fread 오류 발생\n"); } if (fclose(fp) != 0) { perror("fclose 실패"); // 버퍼링된 쓰기 실패도 여기서 드러날 수 있다 }

텍스트 vs 바이너리 모드

Python은 이 구분을 자주 신경 쓰지 않아도 되지만(대부분 인코딩 문제로 국한), C에서는 데이터가 그대로 오가는지 여부가 완전히 달라집니다.

TEXT MODE — "r" / "w" / "a"

개행 변환이 일어날 수 있다

플랫폼에 따라 줄바꿈이 매핑됩니다(예: 윈도우에서 \n ↔ \r\n). 로그, 설정, 소스 코드처럼 사람이 읽는 텍스트에 사용합니다.

BINARY MODE — "rb" / "wb" / "ab"

바이트가 변환 없이 그대로 오간다

구조체를 통째로 덤프하거나 이미지·직렬화 데이터를 다룰 때는 반드시 바이너리 모드를 씁니다.

구조체를 바이너리로 저장할 때의 함정

struct Point { int x, y; }; struct Point p = {3, 4}; FILE *fp = fopen("point.bin", "wb"); fwrite(&p, sizeof p, 1, fp); // 패딩까지 포함해 그대로 기록 fclose(fp); // 주의: 컴파일러/아키텍처마다 구조체 패딩·엔디안이 달라질 수 있어 // 이렇게 만든 파일은 이식성이 없다 (다른 머신에서 못 읽을 수 있음)

strcpy vs strncpy

Python의 문자열은 크기를 몰라도 안전합니다. C에서는 개발자가 목적지 버퍼의 크기를 항상 함께 들고 다녀야 해요.

strcpy

  • 목적지 버퍼 크기를 인자로 받지 않음
  • src가 dest보다 길면 인접 메모리를 덮어씀
  • gets()와 함께 "쓰지 말아야 할 함수"의 대표격
  • 수많은 보안 취약점(CVE)의 원인

strncpy + 직접 널 종료

  • 목적지 크기 n을 항상 함께 전달
  • 단, src 길이가 n 이상이면 널 종료를 보장하지 않는 함정 존재
  • 복사 후 buf[n-1] = '\0'을 직접 붙이는 습관 필요
  • 더 안전하려면 snprintf(buf, n, "%s", src) 패턴도 고려

버퍼 오버플로우와 strncpy의 함정

char dest[8]; // 위험: dest는 8바이트인데 src가 더 길면 스택을 덮어씀 strcpy(dest, "이 문자열은 너무 깁니다"); // strncpy도 완벽하지 않다 — src가 n자 이상이면 널 종료가 안 됨 strncpy(dest, "역시 너무 긴 문자열", sizeof dest); dest[sizeof dest - 1] = '\0'; // 직접 널 종료를 보장

strtok으로 토큰화하기

Python의 "a,b,,c".split(",")는 순수 함수지만, strtok은 원본 문자열을 직접 파괴하고 내부 static 상태를 기억합니다.

strtok 기본 사용

char line[] = "apple,banana,,cherry"; char *tok = strtok(line, ","); while (tok != NULL) { printf("token: %s\n", tok); tok = strtok(NULL, ","); // 두 번째 호출부터는 첫 인자에 NULL } // line 자체가 훼손된다: "apple\0banana\0\0cherry" // -> 원본을 보존해야 하면 strdup 등으로 복사본을 만들어 사용 // -> 연속된 구분자(,,)는 빈 토큰을 만들지 않고 건너뛴다 (Python split과 다름)

sscanf로 문자열 파싱하기

Python의 정규식이나 split() 대신, C는 scanf 계열의 포맷 문자열로 문자열에서 값을 뽑아냅니다. 반환값은 반드시 확인해야 합니다.

필드 개수 체크와 폭 제한

char line[] = "3.14 hello 42"; double f; char word[32]; int n; int matched = sscanf(line, "%lf %31s %d", &f, word, &n); if (matched != 3) { fprintf(stderr, "파싱 실패: %d개 필드만 매칭됨\n", matched); } // %31s: word 버퍼(32바이트)를 넘지 않도록 폭을 지정 // 폭을 안 주면 strcpy처럼 오버플로우 위험이 그대로 생긴다

errno / perror / strerror

Python은 예외가 콜스택을 타고 자동으로 전파됩니다. C는 그런 게 없으니, 실패 직후 errno를 확인하는 습관이 곧 예외 처리입니다.

표준 에러 처리 패턴

#include <errno.h> #include <string.h> FILE *fp = fopen("no_such_file.txt", "r"); if (fp == NULL) { // perror: errno를 사람이 읽을 문구로 바로 stderr에 출력 perror("fopen"); // strerror: 같은 메시지를 문자열로 받아 직접 다루고 싶을 때 fprintf(stderr, "errno=%d (%s)\n", errno, strerror(errno)); return 1; } // 주의: errno는 함수 성공 시 자동으로 0으로 리셋되지 않는다 // -> 반환값으로 실패를 먼저 확인한 뒤에만 errno를 읽을 것

표준 라이브러리 훑기

Python의 배터리 포함(batteries included) 철학과 달리, C 표준 라이브러리는 얇습니다. 이 세 헤더가 실무에서 가장 자주 쓰입니다.

string.h

문자열 · 메모리 조작

strlen, strcmp, strcat, memcpy, memset 등. 대부분 널 종료('\0') 규약을 전제로 동작합니다.

stdlib.h

메모리 할당 · 형변환 · 정렬

malloc/free, atoi/strtol, qsort, exit 등. atoi는 에러를 알려주지 않으니 strtol을 더 선호합니다.

ctype.h

문자 분류 · 변환

isalpha, isdigit, isspace, tolower 등. int로 캐스팅된 unsigned char를 넘겨야 안전합니다.

실습 미리보기: 단어 빈도수 세기

파일을 한 줄씩 읽고, strtok으로 단어를 분리하고, 소문자로 정규화한 뒤 테이블에 누적합니다. Python의 Counter(text.split())를 손수 구현하는 셈입니다.

핵심 루프 스케치

FILE *fp = fopen(argv[1], "r"); if (!fp) { perror("fopen"); return 1; } char line[256]; while (fgets(line, sizeof line, fp)) { char *tok = strtok(line, " \t\n.,!?"); while (tok) { for (char *p = tok; *p; p++) *p = (char)tolower((unsigned char)*p); count_word(tok); // 배열/해시 테이블에 빈도 누적 tok = strtok(NULL, " \t\n.,!?"); } } fclose(fp); print_top_words(); // count_word 구현 선택지: struct { char word[32]; int count; } 배열을 // 선형 탐색하거나, 직접 해시 테이블을 만들어 조회 비용을 낮출 수 있다

흔한 실수

Python에서는 런타임 예외가 즉시 알려주지만, C에서는 조용히 잘못된 메모리를 건드리고 넘어갈 수 있습니다.

01
strcpy로 버퍼 오버플로우 목적지 크기 확인 없이 strcpy(dest, src)를 호출하면 스택/힙이 손상되고, 크래시나 보안 취약점으로 이어질 수 있다. snprintf나 크기를 명시하는 복사 패턴으로 대체할 것.
02
strtok의 재진입 문제 strtok은 내부 static 상태를 공유하므로, 중첩 호출(다른 함수 안에서 strtok을 또 호출)이나 멀티스레드 환경에서 동시에 쓰면 상태가 서로 꼬인다. POSIX의 strtok_r로 대체하는 것이 안전하다.
03
fopen 실패 체크 생략 NULL 체크 없이 곧장 fread/fwrite를 호출하면 NULL 포인터 역참조로 세그폴트가 난다. 반환값은 예외처럼 자동으로 알려주지 않으므로 항상 그 자리에서 확인해야 한다.
