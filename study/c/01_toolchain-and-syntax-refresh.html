컴파일 과정

소스가 실행 파일이 되기까지

Python은 인터프리터가 즉시 실행하지만, C는 gcc 한 번의 명령 뒤에 네 단계가 순서대로 숨어 있어요. 각 단계는 독립적으로 멈춰서 결과물을 확인할 수 있습니다.

STEP 1

전처리

cpp가 #include를 실제 내용으로, #define을 값으로 치환. 결과는 .i

STEP 2

컴파일

전처리된 C 코드를 어셈블리로 변환. gcc -S로 .s 파일만 뽑아볼 수 있음

STEP 3

어셈블

어셈블리를 기계어 목적 파일로 변환. gcc -c 결과가 .o

STEP 4

링킹

여러 .o와 라이브러리를 하나로 묶어 실행 파일 생성. 이때 심볼이 해결됨

gcc / clang

자주 쓰는 플래그, 어느 단계에 관여하나

플래그마다 컴파일 파이프라인의 특정 단계에 개입해요. 어느 단계인지 알면 에러 메시지가 링크 에러인지 컴파일 에러인지 바로 구분할 수 있습니다.

-c

목적 파일까지만

링킹을 생략하고 .o까지만 생성. 여러 파일을 나눠 컴파일할 때 필수.

-o

출력 파일명 지정

gcc a.c -o a.out처럼 결과물 이름을 지정. 안 쓰면 기본값 a.out.

-I

헤더 검색 경로

-Iinclude/ 처럼 #include "..."를 찾을 디렉터리를 전처리 단계에 추가.

-L

라이브러리 경로

-Llib/ 처럼 링킹 단계에서 .a/.so 파일을 찾을 디렉터리를 추가.

-l

라이브러리 링크

-lm처럼 lib 접두사/확장자를 뺀 이름으로 링크. 수학 함수엔 -lm 필수.

-Wall -Wextra

경고 최대로

기본 설정으론 조용히 넘어가는 버그를 잡아준다. 항상 켜고 시작하는 습관.

Makefile

타겟 : 의존성 → 명령

Python엔 빌드 단계가 없어서 낯설 수 있어요. make는 파일의 수정 시각을 비교해서 변경된 것만 다시 컴파일해주는 도구입니다.

Makefile

CC = gcc CFLAGS = -Wall -Wextra -Iinclude # 타겟: 의존성 (탭으로 시작하는 다음 줄이 명령) main: main.o utils.o $(CC) $(CFLAGS) -o main main.o utils.o main.o: main.c utils.h $(CC) $(CFLAGS) -c main.c utils.o: utils.c utils.h $(CC) $(CFLAGS) -c utils.c clean: rm -f *.o main

Makefile

자동 변수와 패턴 규칙

같은 규칙을 파일마다 반복해서 쓰지 않으려고 make가 제공하는 축약 문법이에요. $@ / $< 조합을 알면 Makefile이 훨씬 짧아집니다.

$@

현재 타겟 이름

main.o: main.c 규칙에서 $@는 main.o를 가리킨다.

$<

첫 번째 의존성

main.o: main.c utils.h에서 $<는 main.c(첫 의존성)만 가리킨다.

$^

모든 의존성

의존성 전체를 공백으로 나열. main: main.o utils.o에서 $^는 두 파일 다.

%.o: %.c

패턴 규칙

확장자 규칙을 일반화. .c 파일마다 규칙을 따로 쓰지 않아도 된다.

패턴 규칙로 줄인 버전

%.o: %.c $(CC) $(CFLAGS) -c $< -o $@

프로젝트 구조

.h는 인터페이스, .c는 구현

Python의 모듈은 파일 하나가 곧 네임스페이스지만, C는 헤더(선언)와 소스(정의)가 분리돼요. 각 .c 파일은 독립적으로 컴파일되는 "번역 단위(translation unit)"입니다.

디렉터리 구조

project/ ├── include/ │ └── utils.h // 함수 선언만 (인터페이스) ├── src/ │ ├── main.c // utils.h를 include해서 사용 │ └── utils.c // utils.h를 include해서 구현 └── Makefile

include/utils.h

#ifndef UTILS_H #define UTILS_H int add(int a, int b); // 선언만, 몸체 없음 #endif

문법 재정비

Python/Java와 다른 지점 위주로

기초 문법은 이미 알고 있다고 가정하고, 감각이 다른 부분만 짚어봅니다.

타입

고정폭 정적 타입

int/short/long은 크기가 플랫폼마다 다를 수 있음. 정수 나눗셈은 소수점을 버림(7/2 == 3). 암묵적 str+int 결합 없음.

제어문

elif 없음, switch는 fallthrough

else if만 존재. switch는 break 없으면 다음 case로 흘러 내려감(의도적 fallthrough도 흔함).

함수

사용 전 선언(프로토타입) 필요

기본값 인자·오버로딩 없음. 헤더에 시그니처를 먼저 선언해야 다른 파일에서 호출 가능.

스코프

블록 스코프 + static

{ } 블록마다 스코프. static 지역 변수는 함수 호출 간 값을 유지, static 전역은 그 파일 안에서만 보임.

코드 예제

static 지역 변수, 함수 안의 숨은 상태

Python엔 없는 개념이에요. 지역 변수인데도 함수가 끝나도 값이 사라지지 않고, 다음 호출에서 이어서 사용됩니다.

counter.c

#include <stdio.h> int next_id(void) { static int counter = 0; // 최초 1회만 초기화, 이후 값 유지 return ++counter; } int main(void) { printf("%d\n", next_id()); // 1 printf("%d\n", next_id()); // 2 printf("%d\n", next_id()); // 3 return 0; }

비교

인터프리터 감각으로 보면 낯선 것들

Python

  • 소스를 한 줄씩 읽어 즉시 실행
  • 별도 컴파일·링크 단계 없음
  • import 시점에 모듈을 로드
  • 타입·문법 오류도 실행 중에야 드러남

C

  • 전처리 → 컴파일 → 어셈블 → 링킹 순서로 미리 빌드
  • #include는 텍스트 그대로 붙여넣는 치환
  • 여러 .o 파일을 링커가 하나로 묶어야 실행 가능
  • 대부분의 타입·문법 오류를 컴파일 시점에 발견

실습

멀티파일 프로젝트를 Makefile로 빌드하기

math_utils, string_utils 두 모듈로 나눈 소스를 Makefile 하나로 빌드해봅니다.

01
math_utils.h/.c, string_utils.h/.c, main.c로 분리 각 모듈은 헤더(선언)와 소스(구현)로 나눈다.
02
각 헤더에 헤더 가드 작성 #ifndef/#define/#endif로 중복 include를 막는다.
03
Makefile에 타겟/의존성/패턴 규칙 작성 %.o: %.c 패턴 규칙과 $@/$<로 반복을 줄인다.
04
make, make clean 동작 확인 -Wall -Wextra 경고를 0개로 만든 뒤 완료.

흔한 실수

멀티파일 프로젝트에서 자주 걸리는 것들

01
헤더 가드 빠뜨림 같은 헤더가 여러 경로로 두 번 include되면 "redefinition" 에러. #ifndef/#define/#endif로 방지.
02
include 순환 참조 a.h가 b.h를, b.h가 a.h를 include하면 무한 전개. 헤더 가드 + 전방 선언(struct Foo;)으로 끊는다.
03
-Wall 안 켜서 놓치는 경고 초기화 안 된 변수, 타입 불일치, 사용하지 않는 값 등을 기본 옵션에선 조용히 넘어간다. 처음부터 켜고 개발한다.

아래 질문에 스스로 답해보면서 1주차 내용을 점검해보세요.

전처리 → 컴파일 → 어셈블 → 링킹, 네 단계를 순서대로 설명할 수 있나요?

gcc -E / -S / -c 로 각 단계 결과물을 직접 뽑아볼 수 있어요.

-c, -I, -l 플래그가 각각 어느 단계에 관여하는지 아나요?

-c는 컴파일까지, -I는 전처리, -l/-L은 링킹 단계예요.

Makefile에서 $@ 와 $< 의 차이를 설명할 수 있나요?

$@는 타겟 자신, $<는 의존성 목록의 첫 번째예요.

헤더 가드가 왜 필요한지, 없으면 무슨 일이 벌어지는지 아나요?

같은 선언이 한 번역 단위에 두 번 들어가면 redefinition 에러가 나요.
