모듈 문법

ESM vs CommonJS

JS에는 두 가지 모듈 시스템이 공존합니다. ESM은 최신 표준(브라우저·Node 모두 지원), CommonJS는 Node.js가 오래 써온 방식입니다.

ESM (import/export)

// math.js export function add(a, b) { return a + b; } export const PI = 3.14; // main.js import { add, PI } from "./math.js"; console.log(add(1, 2));

CommonJS (require)

// math.js function add(a, b) { return a + b; } module.exports = { add, PI: 3.14 }; // main.js const { add, PI } = require("./math"); console.log(add(1, 2));

동작 차이

둘은 동작 방식도 다릅니다

CommonJS

  • Node.js 기본값 (.cjs, 또는 type 생략 시)
  • require()는 동기적으로 즉시 로드
  • 브라우저에서 그대로 못 씀 (번들러 필요)
  • module.exports 하나의 객체로 내보냄

ESM

  • 공식 JS 표준, 브라우저 <script type="module"> 지원
  • import는 정적 분석 가능 (트리 쉐이킹에 유리)
  • 파일 확장자 .js 명시 필요 (Node 기준)
  • package.json의 "type": "module"로 활성화

최근 신규 프로젝트는 대부분 ESM을 기본으로 채택합니다.

패키지 메타데이터

package.json이 곧 프로젝트 명세서

Python의 pyproject.toml처럼, package.json은 프로젝트 이름·버전· 의존성·실행 스크립트를 한 파일에 담습니다.

package.json

{ // 이름과 버전 "name": "my-app", "version": "1.0.0", "type": "module", // npm run dev / npm run build 로 실행 "scripts": { "dev": "vite", "build": "vite build" }, // 런타임에 필요한 라이브러리 "dependencies": { "lodash": "^4.17.21" }, // 개발 중에만 필요한 도구 "devDependencies": { "vite": "^5.0.0" } }

생태계 비교

npm ↔ pip / poetry

역할은 거의 동일합니다. 이름과 파일 위치만 다르다고 생각하면 이해가 빠릅니다.

Python

  • pip install requests
  • requirements.txt / pyproject.toml
  • venv, poetry로 가상환경 격리
  • site-packages에 전역 or 가상환경 단위 설치

JavaScript

  • npm install axios
  • package.json / package-lock.json
  • 격리 없이 프로젝트별 node_modules 폴더
  • node_modules에 프로젝트 로컬 설치가 기본

설치와 버전 관리

node_modules와 시맨틱 버전

npm install을 하면 node_modules 폴더에 의존성이 통째로 내려받아집니다. .gitignore에 넣고, package-lock.json으로 정확한 버전을 고정합니다.

node_modules

로컬 설치 폴더

Python의 site-packages와 유사하나 프로젝트마다 독립적으로 생성됨. 용량이 커서 git에 커밋하지 않음.

semver

^4.17.21 같은 표기

MAJOR.MINOR.PATCH. ^는 마이너까지, ~는 패치까지 자동 업데이트 허용을 의미.

번들러가 필요한 이유

브라우저는 npm을 모릅니다

import "lodash" 같은 코드를 브라우저에 그대로 던지면 실패합니다. 브라우저는 node_modules라는 개념도, npm 패키지 이름 해석 방식도 알지 못하기 때문입니다.

1
수백 개의 파일을 그대로 요청할 수 없다 모듈 하나하나가 개별 네트워크 요청이 되면 로딩이 매우 느려짐
2
패키지 이름을 실제 경로로 바꿔줘야 한다 import "lodash" → node_modules/lodash/... 경로로 해석 필요

번들러 = 여러 모듈 파일을 분석해서 브라우저가 읽을 수 있는 하나(또는 몇 개)의 파일로 묶어주는 도구

번들러 둘러보기

Vite와 webpack

둘 다 번들러지만 접근 방식과 개발 경험이 다릅니다.

Vite

빠른 개발 서버 우선

개발 중엔 ESM을 브라우저에 그대로 서빙해 즉시 반영, 배포 시에만 번들링. 최근 신규 프로젝트의 사실상 기본값.

webpack

설정 기반의 오래된 표준

세밀한 설정과 방대한 플러그인 생태계. 레거시 대형 프로젝트에서 여전히 널리 쓰임.

트랜스파일 도구

Babel과 TypeScript는 왜 쓰나

둘 다 "다른 코드를 오늘 브라우저가 이해하는 JS로 변환"한다는 점에서 같은 카테고리(트랜스파일러)입니다.

Babel

최신 문법 → 구형 JS

옵셔널 체이닝 같은 최신 문법을 오래된 브라우저도 실행 가능한 코드로 변환. 오래된 사용자층 지원이 목적.

TypeScript

타입이 있는 JS → JS

컴파일 시점에 타입을 검사한 뒤 타입 정보를 제거한 순수 JS를 출력. Python의 타입 힌트 + mypy를 언어 차원에 통합한 느낌.

흔한 실수

모듈 관련 자주 겪는 에러

01
import와 require를 한 파일에 섞어 쓰다 에러 ESM 환경에서 require is not defined, 또는 그 반대 SyntaxError 발생
02
package.json에 "type" 필드를 설정하지 않음 기본값은 CommonJS이므로 import 문이 그대로 실패하거나 의도와 다르게 해석됨
03
import 경로에 확장자를 생략 Node ESM은 번들러와 달리 .js 확장자 생략을 허용하지 않아 모듈 해석 에러 발생
