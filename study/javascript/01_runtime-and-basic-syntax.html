Runtime

JS가 돌아가는 두 무대

같은 JavaScript 문법이지만, 실행되는 환경에 따라 쓸 수 있는 것이 다릅니다.

브라우저 콘솔

  • 웹 페이지 안에서 실행
  • 전역 객체는 window
  • DOM/BOM 조작 가능
  • F12 → Console 탭에서 접근
  • 파일 시스템 접근 불가(샌드박스)

Node.js

  • OS 위에서 독립적으로 실행
  • 전역 객체는 global(globalThis)
  • DOM 없음, 서버·CLI·빌드용
  • 터미널의 node 명령/REPL로 접근
  • fs, 네트워크 등 시스템 자원 접근 가능

Python으로 치면 브라우저 콘솔은 "웹페이지 안의 인터프리터", Node.js는 python 명령어 그 자체에 가깝습니다.

Variables

변수 선언, 세 가지

Python엔 변수 선언 키워드 자체가 없었지만, JS는 셋 중 하나를 반드시 골라야 합니다.

Legacy

var

함수 스코프. 재선언·재할당 모두 가능. 호이스팅됨. 오늘날은 거의 쓰지 않음.

Mutable

let

블록 스코프. 재할당은 가능하지만 재선언은 불가. 값이 바뀌는 변수에 사용.

Default choice

const

블록 스코프. 재할당 불가(재바인딩 금지, 객체 내부는 변경 가능). 기본으로 우선 선택.

Scope · JS-specific

함수 스코프 vs 블록 스코프

Python엔 이런 구분이 아예 없습니다(if/for는 스코프를 만들지 않음). JS에서는 var만 예외적으로 블록을 무시합니다.

Example

// var: 함수 스코프 — if 블록을 무시하고 새어 나온다 function testVar() { if (true) { var x = 1; } console.log(x); // 1 — 블록 밖인데도 접근됨 } // let: 블록 스코프 — if 블록 안에 갇힌다 function testLet() { if (true) { let y = 1; } console.log(y); // ReferenceError: y is not defined }

Hoisting · JS-specific

var는 끌어올려진다

var 선언은 스코프 최상단으로 끌어올려져(hoisting) undefined로 미리 초기화됩니다. let/const는 TDZ(Temporal Dead Zone)로 접근 자체를 막습니다.

Example

console.log(a); // undefined — 에러가 아니다! var a = 5; console.log(b); // ReferenceError: Cannot access 'b' before initialization let b = 5;

Python이었다면 두 경우 모두 NameError였을 상황입니다. var의 "조용한 undefined"가 버그의 씨앗이 됩니다.

Comparison · JS-specific

== 와 ===

Python의 ==는 타입까지 고려해 비교하지만, JS의 ==는 타입을 강제 변환한 뒤 비교합니다. 실무에서는 항상 ===를 씁니다.

coercion

"5" == 5

true — 문자열이 숫자로 변환된 뒤 비교됨

strict

"5" === 5

false — 타입이 다르므로 변환 없이 즉시 false

coercion

0 == false

true — boolean이 숫자 0으로 변환됨

coercion

null == undefined

true — 이 둘만 예외적으로 서로 같다고 취급

coercion

[] == false

true — 배열이 ""로, ""가 다시 0으로 변환됨

gotcha

NaN === NaN

false — NaN은 자기 자신과도 같지 않음(Number.isNaN 사용)

typeof · JS-specific

typeof는 문자열을 반환

Python의 type()과 달리 typeof는 타입 이름을 문자열로 돌려줍니다. 그리고 유명한 함정이 하나 있습니다.

typeof "text"

"string"

typeof 5

"number"

typeof true

"boolean"

typeof undefined

"undefined"

typeof function(){}

"function"

typeof null

"object" ⚠️

1995년부터 이어진 언어 버그. null 체크는 반드시 === null로.

Control Flow · 빠르게

제어문은 거의 그대로

Python/Java/C에서 이미 아는 개념입니다. 문법 차이만 짚고 넘어갑니다.

조건문

if / else if / else

Java/C와 동일. 소괄호+중괄호 필수.

반복문

for / while / for...of

for...of가 Python의 for-in 역할.

분기

switch

case마다 break 필요(fall-through 주의).

삼항 연산자

cond ? a : b

Python의 a if cond else b와 동일한 역할.

논리 연산자

&& / || / !

and/or/not 대신 기호 사용. 단축 평가 동일.

nullish

??

null/undefined일 때만 대체값 사용(4주차에 자세히).

Hands-on Preview

타입 강제 변환 실험하기

Node.js REPL(터미널에 node 입력) 또는 브라우저 콘솔에 한 줄씩 쳐보며 규칙을 스스로 정리해 보세요.

REPL

> "5" + 3 '53' // + 는 문자열 연결을 우선한다 > "5" - 3 2 // - 는 문자열을 숫자로 변환 후 계산 > "5" * "2" 10 // 산술 연산자는 항상 숫자로 변환 > true + true 2 // boolean이 1/0으로 변환 > [] + [] '' // 배열이 문자열 ""로 변환

규칙: +는 피연산자 중 하나라도 문자열이면 문자열 연결, 그 외 산술 연산자는 무조건 숫자 변환을 시도합니다.

Common Pitfalls

가장 흔한 실수 4가지

01
var 호이스팅으로 인한 조용한 버그 선언 전에 사용해도 에러 없이 undefined가 나와, 문제를 한참 뒤에야 발견하게 됩니다.
02
== 로 비교하다 생기는 예상 밖의 true "0" == false, [] == false 처럼 직관과 어긋나는 결과. 항상 ===를 기본으로 사용하세요.
03
var를 블록 스코프로 착각 if/for 블록 안에서 var로 선언해도 블록 밖에서 그대로 접근됩니다. let/const로 습관을 바꾸세요.
04
typeof null === "object"를 null 체크에 사용 null 여부 확인은 typeof가 아니라 value === null로 해야 합니다.

Wrap-up

오늘의 핵심 요약

Node.js와 브라우저 콘솔은 같은 언어, 다른 무대
const 기본, 재할당 필요할 때만 let, var는 쓰지 않기
비교는 항상 ===, ==의 타입 강제 변환은 신뢰하지 않기
제어문·연산자는 이미 알던 것 + 기호 몇 개만 새로 익히면 끝
