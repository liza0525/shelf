EXPRESS BASICS

Express로 서버 띄우기

Flask의 app = Flask(__name__)나 FastAPI의 app = FastAPI()처럼, Express도 인스턴스 하나로 시작합니다.

SERVER.JS (Express 4.x)

// npm install express const express = require('express'); const app = express(); app.get('/', (req, res) => { res.send('Hello, Express!'); }); app.listen(3000, () => { console.log('서버가 http://localhost:3000 에서 실행 중입니다'); });

ROUTING & MIDDLEWARE

get/post/put/delete와 미들웨어 체인

라우트 핸들러도 사실은 미들웨어입니다. app.use()나 라우트에 등록한 순서대로 요청이 파이프라인처럼 통과합니다.

APP.JS

// 요청 → 미들웨어1 → 미들웨어2 → ... → 핸들러 → 응답 app.use(express.json()); // 모든 요청의 body를 JSON으로 파싱 app.get('/boards', listBoards); app.post('/boards', createBoard); app.put('/boards/:id', updateBoard); app.delete('/boards/:id', deleteBoard); // 라우트별로 미들웨어를 추가로 끼워넣을 수도 있음 app.post('/boards', validateBoard, createBoard);

REQ / RES / NEXT

미들웨어 함수의 세 가지 인자

미들웨어는 (req, res, next) 세 인자를 받는 함수입니다. next()를 호출해야만 다음 단계로 넘어갑니다.

req

요청 객체

Flask의 request와 비슷. req.body, req.params, req.query로 요청 데이터에 접근

res

응답 객체

res.json(), res.status(), res.send()로 응답을 만듦. 한 요청에 응답은 한 번만 보낼 수 있음

next

다음으로 넘기는 함수

next()를 호출해야 다음 미들웨어/핸들러 실행. 안 부르면 요청이 응답 없이 멈춤(hang)

LOGGER MIDDLEWARE

function logger(req, res, next) { console.log(`${req.method} ${req.path}`); next(); // 반드시 호출해야 다음 단계로 진행됨 } app.use(logger); // 모든 라우트보다 먼저 등록

API DESIGN

board CRUD 엔드포인트 설계

인메모리 배열(또는 JSON 파일)을 저장소로 써서 board API 네 가지 엔드포인트를 구성합니다.

POST /boards

Create

req.body에서 title/content를 꺼내 배열에 push, 생성된 게시글을 응답

GET /boards/:id

Read

req.params.id로 배열에서 찾기, 없으면 404 반환

PUT /boards/:id

Update

id로 찾은 뒤 title/content 덮어쓰기, 없으면 404 반환

DELETE /boards/:id

Delete

id로 찾아 배열에서 제거, 204 No Content 응답

VALIDATION

요청 검증 미들웨어 만들기

title/content가 없으면 굳이 저장소까지 가지 않고, 미들웨어 단계에서 걸러냅니다.

VALIDATE-BOARD.JS

function validateBoard(req, res, next) { const { title, content } = req.body; if (!title || typeof title !== 'string') { return res.status(400).json({ error: 'title은 필수 문자열입니다' }); } if (!content || typeof content !== 'string') { return res.status(400).json({ error: 'content는 필수 문자열입니다' }); } next(); // 검증을 통과했을 때만 다음 단계로 } app.post('/boards', validateBoard, createBoard); app.put('/boards/:id', validateBoard, updateBoard);

res.status().json()을 호출할 땐 앞에 return을 꼭 붙이세요 — 안 그러면 next()까지 같이 실행돼 "Cannot set headers after they are sent" 에러가 납니다.

ERROR HANDLING

인자 4개짜리 에러 핸들링 미들웨어

Express는 함수의 매개변수 개수로 에러 미들웨어를 구분합니다. (err, req, res, next) — 정확히 4개여야 합니다.

라우트에서 에러 던지기

app.get('/boards/:id', (req, res, next) => { const board = boards.find(b => b.id === Number(req.params.id)); if (!board) { const err = new Error('게시글을 찾을 수 없습니다'); err.status = 404; return next(err); // 에러 핸들링 미들웨어로 점프 } res.json(board); });

ERROR-HANDLER.JS (반드시 라우트 등록 다음, 맨 마지막에)

app.use((err, req, res, next) => { console.error(err.stack); const status = err.status || 500; res.status(status).json({ error: err.message || 'Internal Server Error' }); });

ASYNC ERRORS

비동기 핸들러에서 에러를 놓치지 않기

async 함수 안에서 발생한 reject는 Express 4.x가 자동으로 못 잡습니다. try/catch로 감싸고 next(err)를 직접 호출해야 합니다.

BAD — 에러를 놓치는 코드 (Express 4.x)

app.get('/boards/:id', async (req, res) => { const board = await db.findBoard(req.params.id); // 여기서 reject되면? res.json(board); // 에러 핸들링 미들웨어로 절대 못 감 });

GOOD — try/catch + next(err)

app.get('/boards/:id', async (req, res, next) => { try { const board = await db.findBoard(req.params.id); if (!board) return res.status(404).json({ error: '게시글 없음' }); res.json(board); } catch (err) { next(err); // 에러 핸들링 미들웨어로 전달 } });

Express 5.x(2024년 정식 릴리즈)부터는 async 핸들러가 reject하면 자동으로 next(err)를 호출해줘서 try/catch 없이도 안전합니다. 다만 현업에는 아직 4.x가 더 널리 쓰이고 있어 이 강의는 4.x 문법을 기준으로 진행합니다.

COMPARE

Flask/FastAPI 미들웨어 대비 무엇이 다른가

가장 큰 차이는 "명시적으로 next()를 호출해야 다음으로 넘어간다"는 점입니다.

Flask / FastAPI

  • @app.before_request/@app.middleware가 모든 요청에 자동 적용
  • next() 개념 없이 함수가 끝나면 자동으로 다음 단계
  • FastAPI는 Depends()로 검증·인증을 의존성 주입처럼 처리
  • 에러 처리는 @app.errorhandler(Exception) 데코레이터로 등록

Express

  • app.use()/라우트에 등록한 순서 = 실행 순서
  • next()를 직접 호출해야만 다음 미들웨어로 넘어감
  • 검증은 직접 짠 미들웨어 함수를 라우트에 끼워 넣는 방식이 흔함
  • 에러 처리는 인자 4개짜리 미들웨어를 맨 마지막에 등록

PITFALLS

자주 하는 실수 세 가지

01
async 핸들러의 에러를 안 잡음 try/catch 없이 await가 reject되면 응답이 영원히 안 가거나, unhandled rejection으로 프로세스가 죽을 수 있다. 항상 try/catch + next(err)로 감싸기
02
미들웨어 등록 순서 실수 에러 핸들링 미들웨어를 라우트보다 먼저 등록하면 절대 호출되지 않는다. express.json()도 라우트보다 앞에 등록해야 req.body가 채워진다
03
응답 후 return을 빼먹음 res.json()을 호출하고도 return을 안 하면 다음 코드가 계속 실행돼 "Cannot set headers after they are sent" 에러가 난다
