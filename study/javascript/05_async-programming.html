History · 01

콜백 지옥, Callback Hell

JS는 원래 콜백 함수로 비동기 작업을 처리했습니다. 작업이 늘어날수록 코드는 오른쪽으로 계속 들여쓰기됩니다.

callback-hell.js

// 사용자 → 주문 → 배송 정보를 순서대로 조회 getUser(userId, function (user) { getOrders(user.id, function (orders) { getShipping(orders[0].id, function (shipping) { console.log(shipping.status); // 콜백 안에 콜백... 에러 처리도 매 단계 반복 }, onError); }, onError); }, onError);

History · 02

Promise로 체인 연결하기

Promise는 "미래에 완료될 값"을 객체로 감싸서 .then()으로 이어붙일 수 있게 합니다. 들여쓰기가 옆이 아니라 아래로 흐릅니다.

promise-chain.js

getUser(userId) .then((user) => getOrders(user.id)) .then((orders) => getShipping(orders[0].id)) .then((shipping) => console.log(shipping.status)) .catch((err) => console.error(err)); // 에러 처리가 .catch() 하나로 통합됨

History · 03

async/await로 동기처럼 읽기

async/await는 Promise 위에 얹힌 문법 설탕(syntactic sugar)입니다. 내부적으로는 여전히 Promise가 동작합니다.

async-await.js

async function loadShipping(userId) { try { const user = await getUser(userId); const orders = await getOrders(user.id); const shipping = await getShipping(orders[0].id); console.log(shipping.status); } catch (err) { console.error(err); } }

Event Loop

콜스택부터 큐까지

JS는 싱글 스레드입니다. 콜스택이 비워질 때마다 이벤트 루프가 큐에서 다음 실행할 작업을 꺼내옵니다.

STEP 1

콜스택

동기 코드가 먼저 실행되며 쌓였다가 순서대로 빠짐

STEP 2

마이크로태스크 큐

Promise.then, async 함수의 재개 — 콜스택이 빌 때마다 큐가 빌 때까지 우선 처리

STEP 3

태스크 큐(매크로)

setTimeout, DOM 이벤트 — 마이크로태스크 큐가 빈 뒤 하나씩 처리

Event Loop · 실행 순서

이 코드는 어떤 순서로 찍힐까?

event-loop-order.js

console.log("1"); setTimeout(() => console.log("2"), 0); Promise.resolve().then(() => console.log("3")); console.log("4"); // 실행 결과: 1 → 4 → 3 → 2 // 동기 코드가 먼저, 그다음 마이크로태스크(Promise), 마지막 매크로태스크(setTimeout)

Python vs JavaScript

asyncio와 이벤트 루프의 차이

Python asyncio

  • asyncio.run()으로 루프를 명시적으로 시작해야 함
  • 코루틴(async def)은 await 하지 않으면 실행되지 않음
  • 동기 코드와 섞으려면 루프를 직접 관리해야 함
  • 멀티스레드/멀티프로세스와 별도로 병행 가능

JavaScript

  • 이벤트 루프가 런타임(브라우저/Node)에 기본 내장됨
  • 따로 시작할 필요 없이 스크립트 실행과 동시에 동작
  • 모든 코드가 싱글 스레드 위에서 이벤트 루프를 공유
  • 진짜 병렬 실행이 필요하면 Worker 등 별도 수단 필요

병렬 처리

Promise.all vs Promise.race

all은 모두 끝날 때까지 기다리고, race는 가장 먼저 끝난 하나만 취합니다.

promise-all-race.js

const results = await Promise.all([ fetchUser(1), fetchUser(2), fetchUser(3), ]); // 셋 다 끝나야 배열로 반환. 하나라도 reject되면 즉시 전체 reject const fastest = await Promise.race([ fetchFromServerA(), fetchFromServerB(), ]); // 둘 중 먼저 끝나는(성공이든 실패든) 하나만 반환

에러 처리

try/catch로 감싸기

async 함수 안에서는 try/catch가 Promise의 .catch()와 동일하게 동작합니다.

error-handling.js

async function loadData() { try { const res = await fetch("/api/data"); if (!res.ok) throw new Error(`HTTP ${res.status}`); return await res.json(); } catch (err) { console.error("불러오기 실패:", err.message); return null; } } // then 체이닝이라면 .catch()로 동일하게 처리 fetch("/api/data").then(handleData).catch(onError);

실습 미리보기

fetch로 여러 API 병렬 호출

여러 사용자 정보를 동시에 조회해서, 전체가 끝나면 한 번에 출력합니다.

fetch-parallel.js

async function fetchUsers(ids) { const requests = ids.map((id) => fetch(`https://api.example.com/users/${id}`).then((res) => res.json()) ); try { const users = await Promise.all(requests); users.forEach((u) => console.log(u.name)); } catch (err) { console.error("일부 요청 실패:", err); } } fetchUsers([1, 2, 3]);

흔한 실수

이런 실수, 조심하세요

01

await를 빠뜨리기 await 없이 async 함수를 호출하면 결과값이 아니라 Promise 객체 자체가 반환됩니다. 콘솔에 [object Promise]가 찍히는 이유입니다.

02

Promise.all은 하나만 실패해도 전체 reject 요청 중 하나라도 실패하면 나머지 결과는 버려지고 catch로 바로 넘어갑니다. 부분 실패를 허용하려면 Promise.allSettled를 사용하세요.

03

순차 await로 병렬성 날리기 서로 의존하지 않는 요청을 for 루프에서 하나씩 await하면 직렬로 실행됩니다. map으로 먼저 시작하고 Promise.all로 모아야 진짜 병렬입니다.

Summary

5주차 핵심 요약

비동기 패턴은 콜백 → Promise → async/await 순으로 진화했다

async/await는 결국 Promise 위에 얹힌 문법일 뿐이다

이벤트 루프는 콜스택 → 마이크로태스크 큐 → 태스크 큐 순으로 돈다

Promise 콜백이 setTimeout보다 항상 먼저 실행된다

JS의 이벤트 루프는 런타임에 기본 내장, asyncio는 명시적으로 실행

await만 해도 알아서 스케줄링된다

진짜 병렬 처리는 map + Promise.all, 순차 await는 병렬이 아니다

에러 처리는 try/catch 또는 .catch()로 반드시 감싸기
