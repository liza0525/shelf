Object Literal · 객체 리터럴

객체는 { } 로 바로 만든다

Python의 dict와 비슷해 보이지만, JS 객체 리터럴은 처음부터 메서드를 품고 있고, 실은 그 뒤에 프로토타입이라는 구조가 숨어 있다.

// 객체 리터럴 — 값과 동작을 한 번에 담는다 const user = { name: "김코딩", age: 28, greet() { // 메서드 축약 문법 return `안녕, ${this.name}`; }, }; console.log(user.name); // "김코딩" console.log(user.greet()); // "안녕, 김코딩"

Prototype Chain · 프로토타입 체인

모든 객체는 다른 객체를 가리키고 있다

user에 없는 프로퍼티를 찾으면, JS 엔진은 포기하지 않고 프로토타입을 타고 올라가며 찾는다. 이 연결고리가 프로토타입 체인이다.

내 객체

user

own property: name, age

프로토타입

Object.prototype

toString, hasOwnProperty 등 공통 메서드

체인의 끝

null

더 이상 찾을 곳이 없다

Prototype Chain · 직접 확인

getPrototypeOf와 __proto__

const user = { name: "김코딩" }; Object.getPrototypeOf(user) === Object.prototype; // true user.__proto__ === Object.prototype; // true (레거시 접근자) user.hasOwnProperty("name"); // true — user 자신의 프로퍼티 user.hasOwnProperty("toString"); // false — 프로토타입에서 상속 user.toString(); // "[object Object]" — 체인을 타고 찾아낸다

__proto__는 과거 호환을 위해 남은 접근자일 뿐, 실무에서는 Object.getPrototypeOf / setPrototypeOf를 쓴다.

Object.create · 프로토타입 상속

class 없이 상속 구조 만들기

// animalProto를 프로토타입으로 삼는 새 객체를 만든다 const animalProto = { speak() { return `${this.name}가 소리를 낸다`; }, }; const dog = Object.create(animalProto); dog.name = "바둑이"; dog.speak(); // "바둑이가 소리를 낸다" Object.getPrototypeOf(dog) === animalProto; // true

Object.create(proto)는 지정한 객체를 프로토타입으로 삼는 새 객체를 만든다. class 없이도 상속은 이렇게 이미 가능하다.

class · 문법 설탕

class Animal {}을 뜯어보기

class Animal { constructor(name) { this.name = name; } speak() { return `${this.name}가 소리를 낸다`; } } typeof Animal; // "function" — class는 사실 함수다 typeof Animal.prototype.speak; // "function" — speak는 prototype에 붙어있다 const a = new Animal("바둑이"); Object.getPrototypeOf(a) === Animal.prototype; // true

class 선언은 함수(생성자) + prototype에 메서드 등록을 자동으로 해주는 문법일 뿐, 완전히 새로운 실행 모델이 아니다.

extends · 상속 증명

extends도 체인을 연결할 뿐이다

class Dog extends Animal { bark() { return `${this.name}: 멍멍!`; } } const d = new Dog("바둑이"); d.speak(); // Animal.prototype에서 물려받음 d.bark(); // Dog.prototype 자기 것 Object.getPrototypeOf(Dog.prototype) === Animal.prototype; // true d instanceof Animal; // true

extends는 Dog.prototype의 프로토타입을 Animal.prototype으로 연결해줄 뿐이다. 05번 슬라이드의 Object.create와 원리가 같다.

this · 4가지 규칙

this는 "누가 호출했는가"로 결정된다

1. 일반 함수 호출

func()

this는 전역 객체(strict 모드에서는 undefined)를 가리킨다.

2. 메서드 호출

obj.method()

this는 obj — 점(.) 바로 앞의 객체가 된다.

3. 생성자 호출

new Func()

this는 new가 새로 만드는 인스턴스가 된다.

4. call / apply / bind

func.call(obj)

this를 개발자가 명시적으로 지정해서 호출한다.

this · 코드로 확인

같은 함수, 다른 this

function whoAmI() { return this; } whoAmI(); // 1) 일반 호출 → globalThis / undefined const user = { name: "김코딩", whoAmI }; user.whoAmI(); // 2) 메서드 호출 → user 객체 function User(name) { this.name = name; } const u = new User("이코딩"); // 3) 생성자 호출 → 새 인스턴스 whoAmI.call({ name: "박코딩" }); // 4) call → 명시된 객체 whoAmI.apply({ name: "박코딩" }); // apply → 인자를 배열로 전달 const bound = whoAmI.bind({ name: "박코딩" }); bound(); // bind → this가 영구히 고정된 새 함수

Arrow Function · 렉시컬 this

화살표 함수는 자신만의 this가 없다

const timerBroken = { seconds: 0, start() { setInterval(function () { this.seconds++; // 일반 함수 → this가 timerBroken이 아님 console.log(this.seconds); // NaN 혹은 에러 }, 1000); }, }; const timerFixed = { seconds: 0, start() { setInterval(() => { this.seconds++; // 화살표 함수 → 바깥(start)의 this를 그대로 사용 console.log(this.seconds); // 1, 2, 3 ... }, 1000); }, };

화살표 함수는 정의될 때의 스코프에서 this를 "빌려온다"(lexical this). 그래서 call/apply/bind로도 바꿀 수 없다.

Python vs JavaScript

self는 인자, this는 호출의 결과

Python — self

  • 메서드의 명시적인 첫 번째 매개변수
  • def greet(self): 처럼 항상 눈에 보인다
  • instance.method()로 부르면 self는 instance로 고정
  • 정의만 봐도 self가 무엇인지 코드로 드러난다

JavaScript — this

  • 매개변수가 아니라 호출 방식이 결정하는 값
  • 함수 정의만 봐서는 this를 알 수 없다
  • 같은 함수도 호출 방식에 따라 this가 매번 바뀐다
  • 화살표 함수만 예외적으로 정의 시점의 this를 고정한다

실습 · Hands-on

class 없이, class처럼 만들어보기

같은 Counter를 프로토타입 버전과 class 버전으로 각각 만들고, Object.getPrototypeOf로 두 버전의 구조가 결국 같다는 걸 직접 확인한다.

PROTOTYPE 버전

const counterProto = { increment() { this.count++; return this.count; }, }; function createCounter(start) { const counter = Object.create(counterProto); counter.count = start; return counter; }

CLASS 버전

class Counter { constructor(start) { this.count = start; } increment() { this.count++; return this.count; } }

흔한 실수 · Pitfalls

this는 이렇게 자주 사고를 낸다

01

콜백에서 this를 잃어버림 setTimeout, addEventListener 등에 일반 함수를 그대로 넘기면 호출 방식이 바뀌면서 this도 원하던 객체를 더 이상 가리키지 않는다. 화살표 함수나 bind로 고정해야 한다.

02

화살표 함수를 메서드로 정의 const obj = { value: 1, get: () => this.value } 처럼 쓰면 화살표 함수라 this가 obj를 가리키지 않는다. 메서드는 일반 함수(축약 문법)로 정의해야 한다.

03

메서드만 떼어 콜백으로 넘김 onClick={instance.handleClick}처럼 메서드 참조만 넘기면 this가 사라진다. instance.handleClick.bind(instance)나 화살표 함수 프로퍼티로 감싸야 한다.

마무리

class는 겉모습일 뿐, 뼈대는 프로토타입이다

프로토타입 체인과 this의 동적 바인딩은 이후 배열/객체 메서드, 비동기 콜백, DOM 이벤트 핸들러까지 계속 따라다니는 개념이다.

Prototype

객체는 프로토타입으로 연결된다

Object.getPrototypeOf로 체인을 타고 프로퍼티를 찾는다.

class

결국 문법 설탕이다

내부는 함수 생성자 + prototype 메서드 등록일 뿐이다.

this

호출 방식이 this를 결정한다

정의가 아니라 호출 시점에 정해진다. 화살표 함수만 예외다.
